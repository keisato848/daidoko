> 前提資料: `docs/アーキテクチャ設計.md` §4（同期方式の原案）・`docs/データ設計.md`

## 0. 目的と非目的

**目的**: 家族グループ内でレシピ等のデータが「何もしなくても」全端末に行き渡る自動同期。
Android⇄iOS 混在の家族で成立すること（1.5.0 で iOS 初回リリース後の最重要ギャップ）。

**非目的（やらないこと）**:

- メールアドレス・パスワード等の**個人情報を伴うアカウント制**（→ §2 アカウントレス）
- リアルタイム共同編集（WebSocket/CRDT）— ポーリング＋プッシュ通知トリガーで十分
- Memo のマージ UI（原案 §4.2 の Vector Clock 案は **LWW に簡素化**。件数・重要度に対して過剰）
- E2E 暗号化（将来オプション。v2.0 は TLS＋グループトークン分離まで）
- Web 版同期

**設計原則**:

1. **ローカルファースト温存** — SQLite が常に真実。同期は「後ろで走る差分転送」で、オフラインでも全機能が動く現状を壊さない
2. **個人情報ゼロ** — サーバーが持つのは乱数 ID・表示名（任意・ニックネーム）・暗号化された push トークンのみ
3. **コスト上限** — 増分ランニングコストは月 $5 以内（§8）。プッシュ通知は Expo Push で ¥0

## 1. 全体像

```
デバイスA(Android)              Railway サーバー                 デバイスB(iOS)
  SQLite(真実)                  Hono + PostgreSQL                SQLite(真実)
     │ 書き込み時に sync_queue へ積む      │                            │
     │── POST /sync/push ───────────>│ LWW マージ・seq 採番        │
     │<─ {applied, serverSeq} ───────│                            │
     │                               │── Expo Push "変更あり" ──> │ (無料・ベストエフォート)
     │                               │<────── GET /sync/pull?since=seq ──│
     │                               │─ {changes[]} ────────────>│ ローカル適用(LWW)
```

- **push**: ローカル書き込みのたびに `sync_queue` へ積み、オンライン時にまとめて送る（既存設計 §4.3 踏襲）
- **pull**: `前回受信した seq 以降の変更をくれ` 方式（グループ単位の単調増加シーケンス）。起動時・フォアグラウンド復帰時・push 通知受信時に実行
- **通知**: サーバーは変更を受けたら同グループの他端末へ Expo Push を投げる（同期トリガー兼「家族がレシピを追加しました」のお知らせ）

## 2. アカウントレス認証とグループモデル

既存の家族機能（S16: ローカルの家族プロフィール＋招待コード）をそのままクラウドの「同期グループ」に昇格させる。

### 2-1. 識別子

| もの          | 実体                                                     | 保存場所                                                         |
| ------------- | -------------------------------------------------------- | ---------------------------------------------------------------- |
| device_id     | UUID（初回起動時生成）                                   | 端末 SQLite + サーバー                                           |
| device_secret | 乱数 32byte。API 認証に使用                              | **expo-secure-store**（Keystore/Keychain）+ サーバー（ハッシュ） |
| group_id      | UUID                                                     | 端末 + サーバー                                                  |
| invite_code   | 短い英数字（既存 UI の招待コード。**ローテーション可**） | サーバー（有効期限つき）                                         |

### 2-2. フロー

- **グループ作成**: `POST /sync/groups`（device 登録と同時）→ `{group_id, invite_code}`。既存家族画面の「招待コードを共有」がそのまま入口
- **参加**: `POST /sync/groups/join {invite_code}` → 端末がグループに紐づく。以後の API は `Authorization: Bearer <device_id>:<device_secret>`
- **退出/端末除名**: `POST /sync/groups/leave` / オーナー端末からの `DELETE /sync/devices/:id`。除名後は invite_code をローテーションして再入場を防ぐ
- **機種変更**: 既存の移行 ZIP に device 資格情報は**含めない**（secure-store 方針踏襲）。新端末は招待コードで再参加すれば全データが pull で戻る — **移行 ZIP なしの機種変更も成立するようになる**副次効果

### 2-3. 脅威モデル（割り切り）

- 招待コードが漏れると第三者がグループに入れる → 有効期限（例: 72h）＋手動ローテーション＋参加時に全端末へ通知で緩和。家庭内データの機微度に対して十分
- サーバー管理者（=開発者）は平文データを読める → プライバシーポリシーに明記。E2E は将来オプション

## 3. サーバーデータモデル（PostgreSQL 16 + Drizzle）

```
sync_groups   (id, invite_code, invite_expires_at, created_at)
devices       (id, group_id, secret_hash, display_name, expo_push_token, last_pull_seq, created_at)
entities      (group_id, entity_type, entity_id,         -- 複合PK
               payload JSONB, updated_at, deleted_at,
               seq BIGINT,                               -- グループ内単調増加（変更のたび採番）
               updated_by_device)
photos        (id, group_id, r2_key, bytes, created_at)  -- 実体は R2
```

- **最新状態方式**（オペレーションログは持たない）: `entities` は各行の最新版のみ保持。pull は `WHERE group_id=? AND seq > :since ORDER BY seq`
- 削除は `deleted_at` の**トゥームストーン**（クライアントはローカル行を削除）。90 日で物理削除のバッチ
- `payload` はクライアントのテーブル行をそのまま JSONB 化（サーバーはスキーマの意味を知らない＝**サーバー改修なしにアプリ側のカラム追加が可能**）
- インデックス: `(group_id, seq)` と `(group_id, entity_type, entity_id)`

## 4. クライアント側の変更

- `sync_queue` テーブル追加（既存設計 §4.3 の形そのまま: entityType/entityId/operation/payload/retryCount）
- 各サービスの書き込みパスに「queue へ積む」を追加（recipe/tag/shopping/pantry/cooking-log/…。**書き込み API は不変**なので画面側の改修はゼロ）
- `sync.service.ts`: flush（push）・pull・適用（LWW 判定）・NetInfo 監視・フォアグラウンド復帰フック
- 適用時の LWW: 受信 `updated_at` > ローカル `updated_at` なら上書き。同値はサーバー優先。**時計ずれ対策**: push 時にサーバーが `updated_at` を自分の時計で上書きし単調性を保証（クライアント時計を信用しない）
- FTS・作りたい・派生データはローカル適用後に既存のフックで再構築

## 5. エンティティ別戦略（原案 §4.2 を v2.0 スコープに更新）

| エンティティ                                 | 方式                               | 競合解決                                                  | フェーズ |
| -------------------------------------------- | ---------------------------------- | --------------------------------------------------------- | -------- |
| Recipe / Ingredient / Step / Tag / RecipeTag | 差分同期                           | LWW（レシピ単位。材料・手順はレシピの子として丸ごと置換） | **S1**   |
| ShoppingList / ShoppingItem                  | 差分同期                           | LWW（項目単位 — チェック操作が主なので実質競合しない）    | **S2**   |
| PantryItem / NameAlias                       | 差分同期                           | LWW                                                       | **S2**   |
| CookingLog / CookingPhoto / RecipeRevision   | 追記のみ                           | 競合なし（INSERT のみ）                                   | **S3**   |
| Memo                                         | 差分同期                           | LWW（Vector Clock 案は廃止）                              | S3       |
| 写真バイナリ                                 | R2 へ遅延アップロード/ダウンロード | —                                                         | **S3**   |
| users / families（プロフィール）             | 差分同期                           | LWW                                                       | S1       |
| 設定・BYOK キー・AI 利用カウント             | **同期しない**（端末ローカル）     | —                                                         | —        |

## 6. 写真（Cloudflare R2）

- バケット非公開。クライアントは `POST /sync/photos/presign` で PUT/GET の署名付き URL を取得して直接転送（サーバーを太らせない）
- アップロードは Wi-Fi 時優先のバックグラウンドキュー。ダウンロードは表示時に遅延取得＋ローカルキャッシュ
- 既存の圧縮パイプライン（長辺 1600px JPEG q80）をそのまま通すので 1 枚 ≈ 200〜400KB。**無料枠 10GB ≈ 3〜5 万枚**で当面無料。エグレスも R2 は無料

## 7. プッシュ通知（コスト ¥0）

- **Expo Push Notifications** を採用 — アプリは expo-notifications 導入済み（タイマー・残量通知）で、追加 SDK なし・**無料・Android/iOS 両対応**。FCM/APNs 資格情報は EAS が管理
- devices に `expo_push_token` を保存（参加時・トークン変更時に `POST /sync/devices/token`）
- サーバーは push 受領後に同グループの**他**端末へ送信。**5 分デバウンス＋種別ダイジェスト**（「⚡ 家族がレシピを 2 件追加しました」）で通知疲れを防ぐ。買い物リスト変更は即時（買い物中に効くのが価値なので）
- 通知はベストエフォート — 届かなくても次回起動時の pull で追いつく（通知を同期の前提にしない）

## 10. リスクと未決事項

- **無料ユーザーの写真容量**: R2 10GB 超過後は $0.015/GB/月と安価だが、1 グループあたりの写真上限（例: 2,000 枚）をサーバー側に置くのが安全
- **サーバー障害時**: ローカルファーストなのでアプリは動き続ける。queue が溜まるだけ（上限＋古い順破棄を設ける）
- **Expo Push の信頼性**: ベストエフォート前提の設計なので致命的ではない。将来量が増えたら FCM/APNs 直叩きへ移行可能
- **未決**: 招待コードの有効期限の既定値／グループの最大端末数（案: 10）／トゥームストーン保持期間（案: 90 日）— 実装時に確定

## 11. AWS 移行オプションと判断基準（2026-07-13 整理・当面は現行設計で進める）

本設計（Railway PostgreSQL + R2）の代替として **AWS 構成（Lambda + DynamoDB + S3）** を検討した。
`infra/`（CDK: Lambda + Function URL + Budgets/CloudWatch ガード）と `apps/server/src/lambda.ts` が流用できる。
**ユーザー決定: 当面は現行設計。以下の判断基準に該当したら AWS 版を再検討する。**

### 11-1. 比較の要点（2026-07 時点の結論）

| 観点       | 現行（Railway PG + R2） | AWS（Lambda + DynamoDB + S3）                                                                |
| ---------- | ----------------------- | -------------------------------------------------------------------------------------------- |
| 増分コスト | +$3〜6/月               | ≈$0.3〜1/月（常時無料枠は旧アカウントにも適用。AI サーバーも移せば Railway $5 も解約可）     |
| 工数       | 設計済み（5〜6.5 週）   | **+1〜2 週**（DynamoDB シングルテーブル設計への §3 書き直し・Drizzle 不可）                  |
| 副産物     | —                       | レート制限/利用量カウントの共有ストア化（正しくなる）・AWS 実務経験                          |
| リスク     | 低                      | 請求事故（Budgets+同時実行キャップで抑制設計済み）・コールドスタート（同期用途では実害なし） |

### 11-2. AWS 移行を再検討するトリガー（いずれか該当で）

1. **コスト**: Railway 合計（Hobby $5＋PG 従量）が**月 $10 を超えた**（AWS 化でほぼ ¥0 に落とせる差が月 ¥1,500 相当を超える）
2. **スケール**: 同期ユーザー増で Railway PG の使用量課金が伸び始めた（DynamoDB 常時無料 25GB/25RCU/25WCU が効く規模）
3. **厳密なレート制限が必要になった**: in-memory カウンタ（再起動で消える・単一インスタンス前提）では不足する事態（複数インスタンス化・悪用対策の強化）— DynamoDB 共有ストア化とセットで移行が自然
4. **キャリア判断**: AWS の実務経験を職務経歴に積みたいタイミング（転職完了後など） — 学習目的だけでも移行の正当な理由とする

### 11-3. 移行しない・見送る基準

- リリース直前など **+1〜2 週の工数**を割けない時期
- 同期サーバー側に **SQL 前提の要件**（横断集計・複雑な検索・アドホッククエリ）が生まれた場合 — PostgreSQL 継続が有利
- 旧 AWS アカウント（2021 年頃使用）の棚卸しが済んでいない場合 — 移行前にログイン可否・請求履歴 $0 確認・残骸リソースの掃除が前提（または別メールで新規アカウント: $200 クレジット対象）

### 11-4. 移行する場合の原則

- **ハイブリッドで段階移行**: まず同期バックエンド（新規開発）を AWS で構築し、稼働中の AI サーバー（Railway）は触らない → 同期が安定してから AI エンドポイントを Lambda へ移し Railway を解約
- 写真は R2 → S3 に置き換え（presigned URL 方式は共通なのでクライアント改修は最小）
- プッシュ通知（Expo Push）と GAS 利用量アラートは構成に依存しないためそのまま
