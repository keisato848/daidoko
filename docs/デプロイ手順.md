# だいどこ — デプロイ手順

> 対象: v1.0.0 ローカルファースト Android リリース
>
> 方針: v1.0.0 は端末内 SQLite を正とするローカルアプリとして配布・検証する。Railway サーバー、EAS Build、TestFlight、クラウド同期は必須デプロイ対象に含めない。

---

## 1. リリース対象

v1.0.0 の配布対象は Android release APK です。

| 項目            | 値                                                                  |
| --------------- | ------------------------------------------------------------------- |
| Android package | `com.daidoko.app`                                                   |
| versionName     | `1.0.0`                                                             |
| versionCode     | `10000`                                                             |
| APK             | `apps/mobile/android/app/build/outputs/apk/release/app-release.apk` |
| データ保存先    | 端末内 SQLite / アプリ document directory                           |

既存データを残して更新する場合は、必ず `adb install -r` を使います。`adb uninstall`、`pm clear`、Android Studio の Clear data、端末上でのアプリ削除はローカル SQLite と保存写真を消すため実行しません。

---

## 2. 前提環境

- Windows PowerShell
- Node.js 20 以上
- pnpm 9 以上
- JDK 17
- Android SDK Platform-Tools (`adb.exe`)
- USB デバッグを許可済みの Android 実機、または起動済み Android Emulator

推奨の環境変数設定:

```powershell
cd C:\Projects\daidoko

$env:JAVA_HOME = "C:\Program Files\Microsoft\jdk-17.0.19.10-hotspot"
$env:Path = "$env:JAVA_HOME\bin;$env:Path"
$ADB = Join-Path $env:LOCALAPPDATA "Android\Sdk\platform-tools\adb.exe"
$GRADLE_CACHE = Join-Path $env:TEMP "daidoko-gradle-project-cache"
```

依存関係が未インストールの場合:

```powershell
pnpm install --frozen-lockfile
```

## 3. 事前チェック

リリース APK を作る前に、ワークスペースルートで品質チェックを通します。

```powershell
cd C:\Projects\daidoko

pnpm format:check
pnpm lint
pnpm typecheck
pnpm test
```

いずれかが失敗した場合は、APK 作成とタグ付けに進みません。

## 4. Android Release APK のビルド

### 4-1. 実機向け Pixel / arm64-v8a

Pixel 9a などの物理端末で検証する release APK を作ります。

```powershell
cd C:\Projects\daidoko\apps\mobile\android

.\gradlew.bat :app:assembleRelease `
  --project-cache-dir "$GRADLE_CACHE" `
  --no-daemon `
  --console=plain `
  -x lint `
  -x test `
  -x lintVitalAnalyzeRelease `
  -x lintVitalReportRelease `
  -x lintVitalRelease `
  -PreactNativeArchitectures=arm64-v8a
```

### 4-2. Emulator 向け x86_64

Android Emulator で release APK を検証する場合はこちらを使います。

```powershell
cd C:\Projects\daidoko\apps\mobile\android

.\gradlew.bat :app:assembleRelease `
  --project-cache-dir "$GRADLE_CACHE" `
  --no-daemon `
  --console=plain `
  -x lint `
  -x test `
  -x lintVitalAnalyzeRelease `
  -x lintVitalReportRelease `
  -x lintVitalRelease `
  -PreactNativeArchitectures=x86_64
```

APK はどちらも以下に出力されます。

```text
apps/mobile/android/app/build/outputs/apk/release/app-release.apk
```

v1.0.0 のローカル検証 APK は現状 Android debug keystore で署名されています。ストア配布や外部配布へ進める場合は、release signingConfig を専用キーストアへ切り替えてから別途検証します。

## 5. 端末へインストール

### 5-1. 接続確認

```powershell
cd C:\Projects\daidoko

& $ADB devices
```

実機 Pixel 9a の検証済みシリアル:

```text
5B131JEBF16695
```

以降の例では `$DEVICE` に対象端末を指定します。

```powershell
$DEVICE = "5B131JEBF16695"
$APK = "apps/mobile/android/app/build/outputs/apk/release/app-release.apk"
```

### 5-2. データを残して更新

```powershell
& $ADB -s $DEVICE install -r $APK
```

同じ versionCode の APK を入れ直すローカル検証で downgrade 扱いになる場合のみ、次を使います。

```powershell
& $ADB -s $DEVICE install -r -d $APK
```

`INSTALL_FAILED_UPDATE_INCOMPATIBLE` が出た場合は署名が変わっています。データ保持が必要な端末ではアプリ削除で解決せず、同じ署名の APK を使うか、アプリ内バックアップを作成してから移行手順を決めます。

### 5-3. バージョン確認

```powershell
& $ADB -s $DEVICE shell dumpsys package com.daidoko.app |
  Select-String "versionName|versionCode"
```

期待値:

```text
versionCode=10000
versionName=1.0.0
```

## 6. E2E 検証

E2E は release APK をインストールした端末に対して実行します。端末はロック解除済みにし、画面を点灯した状態にします。

### 6-1. 実機 Pixel 9a

```powershell
cd C:\Projects\daidoko

& $ADB -s 5B131JEBF16695 shell cmd statusbar collapse
& $ADB -s 5B131JEBF16695 shell input keyevent KEYCODE_WAKEUP
& $ADB -s 5B131JEBF16695 shell wm dismiss-keyguard

$env:TARGET_DEVICE = "5B131JEBF16695"
$env:ADB_PATH = $ADB
node e2e/android-e2e.mjs
```

v1.0.0 の実機検証結果:

```text
結果: PASS 13 / FAIL 0 / 合計 13
```

検証範囲:

- アプリ起動とホーム描画
- フィルタータブと主要タブ遷移
- 手動レシピ作成、一覧、詳細、削除
- 料理中モードと調理記録写真 UI
- タイムライン表示
- URL 取り込みの応答
- テキスト取り込み AI 指示コピー
- OCR 入口画面
- 設定、バックアップ、ライセンス情報、家族グループ

### 6-2. Emulator

Emulator でも同じ script を使います。`TARGET_DEVICE` は `adb devices` のシリアルへ置き換えます。

```powershell
cd C:\Projects\daidoko

$env:TARGET_DEVICE = "emulator-5554"
$env:ADB_PATH = $ADB
node e2e/android-e2e.mjs
```

スクリーンショットと UI dump は以下に保存されます。

```text
e2e/screenshots/e2e-android
e2e/ui-dumps
```

## 7. 手動スモークチェック

E2E 通過後、必要に応じて次の手動確認を行います。

- 設定 → バックアップ・復元 → バックアップを作成
- 設定 → バックアップ・復元 → 最新バックアップから復元
- 設定 → ライセンス情報 → OSS ライセンス一覧表示
- レシピ詳細 → 調理開始 → 完成して記録 → 写真追加 UI 表示
- アプリ再起動後、既存レシピと調理ログが残っていること

## 8. GitHub 反映とタグ

すべてのチェックが通ったら変更をコミットし、`develop` を push します。

```powershell
cd C:\Projects\daidoko

git status --short
git add <変更ファイル>
git commit -m "..."
git push origin develop
```

v1.0.0 タグを新規作成する場合:

```powershell
git tag -a v1.0.0 -m "v1.0.0"
git push origin v1.0.0
```

既に GitHub 上に存在するタグは、明示的な合意なしに移動・削除・再作成しません。タグ作成後にドキュメントやテスト安定化のコミットを追加した場合は、タグが指すリリース本体のコミットと、追加コミットの目的をリリースメモまたは PR に明記します。

---

## 9. v1.0.0 では不要なデプロイ

次の作業は v1.0.0 の必須デプロイ手順ではありません。

- Railway サーバーの新規作成・更新
- PostgreSQL などの外部 DB 作成
- EAS Build のクラウドビルド
- TestFlight / App Store Connect 提出
- クラウド同期用の認証・API 設定

将来サーバー機能や iOS 配布を有効化する場合は、この手順書とは別に、データ移行、バックアップ、ロールバック、署名、ストア審査を含むデプロイ手順を作成します。

## 10. トラブルシューティング

| 症状                                            | 対処                                                                                                         |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `No authorized device`                          | 端末の USB デバッグ許可を確認し、`adb devices` で `device` 表示になるまで待つ。                              |
| `uiautomator dump` が固まる                     | Node E2E 本体は止めず、固まった `adb ... shell uiautomator dump /sdcard/_e2e.xml` の子プロセスだけ停止する。 |
| 通知シェードやクイック設定が UI dump に混入する | `adb shell cmd statusbar collapse` を実行する。E2E script も通知シェード検出時に collapse して再試行する。   |
| `INSTALL_FAILED_UPDATE_INCOMPATIBLE`            | 署名不一致。データ保持が必要なら uninstall せず、同じ署名の APK で更新する。                                 |
| versionName / versionCode が違う                | `apps/mobile/app.json` と `apps/mobile/android/app/build.gradle` の値を確認し、再ビルドする。                |
| Gradle の `.cxx` 周辺でファイルロックする       | `apps/mobile/android/gradlew.bat --stop` 後、関連 Java プロセスを停止してから再ビルドする。                  |
| 既存データが見えない                            | アプリ削除、`pm clear`、別 package、署名違いによる再インストールが起きていないか確認する。                   |
