# だいどこ — デプロイ手順

> 対象: v1.0.1 ローカルファースト Android リリース
>
> 方針: v1.0.1 は端末内 SQLite を正とし、機種変更用バックアップを ZIP として外部共有できるローカルアプリとして配布・検証する。Railway サーバー、EAS Build、TestFlight、クラウド同期は必須デプロイ対象に含めない。

---

## 1. リリース対象

v1.0.1 のローカル検証・直接配布対象は Android release APK です。Google Play 内部テストへ提出する場合も、同じ versionName の Play 用 AAB を別途ビルドします。

| 項目            | 値                                                                     |
| --------------- | ---------------------------------------------------------------------- |
| Android package | `com.daidoko.app`                                                      |
| versionName     | `1.0.1`                                                                |
| versionCode     | `10003`                                                                |
| APK             | `apps/mobile/android/app/build/outputs/apk/release/app-release.apk`    |
| Play AAB        | `apps/mobile/android/app/build/outputs/bundle/release/app-release.aab` |
| データ保存先    | 端末内 SQLite / アプリ document directory                              |

既存データを残して更新する場合は、必ず `adb install -r` を使います。`adb uninstall`、`pm clear`、Android Studio の Clear data、端末上でのアプリ削除はローカル SQLite と保存写真を消すため実行しません。

---

## 2. 前提環境

- Windows PowerShell
- Node.js 20 以上
- pnpm 9 以上
- JDK 17
- Android SDK Platform-Tools (`adb.exe`)
- USB デバッグを許可済みの Android 実機、または起動済み Android Emulator

推奨の環境変数設定:

```powershell
cd C:\Projects\daidoko

$env:JAVA_HOME = "C:\Program Files\Microsoft\jdk-17.0.19.10-hotspot"
$env:Path = "$env:JAVA_HOME\bin;$env:Path"
$ADB = Join-Path $env:LOCALAPPDATA "Android\Sdk\platform-tools\adb.exe"
$GRADLE_CACHE = Join-Path $env:TEMP "daidoko-gradle-project-cache"
```

依存関係が未インストールの場合:

```powershell
pnpm install --frozen-lockfile
```

`pnpm install` では `expo-modules-core@1.12.26` に対する API 35 向けのローカル patch も自動適用されます。

## 4. Android Release APK のビルド

### 4-1. 実機向け Pixel / arm64-v8a

Pixel 9a などの物理端末で検証する release APK を作ります。

```powershell
cd C:\Projects\daidoko\apps\mobile\android

.\gradlew.bat :app:assembleRelease `
  --project-cache-dir "$GRADLE_CACHE" `
  --no-daemon `
  --console=plain `
  -x lint `
  -x test `
  -x lintVitalAnalyzeRelease `
  -x lintVitalReportRelease `
  -x lintVitalRelease `
  -PreactNativeArchitectures=arm64-v8a
```

### 4-2. Emulator 向け x86_64

Android Emulator で release APK を検証する場合はこちらを使います。

```powershell
cd C:\Projects\daidoko\apps\mobile\android

.\gradlew.bat :app:assembleRelease `
  --project-cache-dir "$GRADLE_CACHE" `
  --no-daemon `
  --console=plain `
  -x lint `
  -x test `
  -x lintVitalAnalyzeRelease `
  -x lintVitalReportRelease `
  -x lintVitalRelease `
  -PreactNativeArchitectures=x86_64
```

APK はどちらも以下に出力されます。

```text
apps/mobile/android/app/build/outputs/apk/release/app-release.apk
```

v1.0.1 のローカル検証 APK は、アップロード署名情報が未設定の場合 Android debug keystore で署名されます。この APK は Google Play へ提出しません。

### 4-3. Google Play 用 AAB

Google Play へ提出する場合は APK ではなく AAB を使います。`bundleRelease` はアップロード署名情報が未設定の場合に失敗するようにしてあります。

`apps/mobile/android` が存在しない、または `apps/mobile/app.json` / `apps/mobile/plugins/withDaidokoOcr.js` を更新した直後は、先に Expo prebuild で native プロジェクトを再生成します。config plugin が `android.compileSdkVersion=35`、`android.targetSdkVersion=35`、`android.suppressUnsupportedCompileSdk=35` を書き込みます。

```powershell
cd C:\Projects\daidoko

pnpm --filter mobile exec expo prebuild --platform android --no-install
```

アップロードキーはリポジトリへコミットしません。環境変数、または `apps/mobile/android/keystore.properties` に次の値を設定します。パスワード類はチャットやログに貼り付けず、ローカル端末上で直接入力・管理します。

```powershell
$env:DAIDOKO_UPLOAD_STORE_FILE = "C:\secure\daidoko-upload.keystore"
$env:DAIDOKO_UPLOAD_STORE_PASSWORD = "<store password>"
$env:DAIDOKO_UPLOAD_KEY_ALIAS = "daidoko-upload"
$env:DAIDOKO_UPLOAD_KEY_PASSWORD = "<key password>"
```

`keystore.properties` を使う場合のキー名も同じです。

```properties
DAIDOKO_UPLOAD_STORE_FILE=C:\secure\daidoko-upload.keystore
DAIDOKO_UPLOAD_STORE_PASSWORD=<store password>
DAIDOKO_UPLOAD_KEY_ALIAS=daidoko-upload
DAIDOKO_UPLOAD_KEY_PASSWORD=<key password>
```

AAB をビルドします。

```powershell
cd C:\Projects\daidoko\apps\mobile\android

.\gradlew.bat :app:bundleRelease `
  --project-cache-dir "$GRADLE_CACHE" `
  --no-daemon `
  --console=plain `
  -x lint `
  -x test
```

出力:

```text
apps/mobile/android/app/build/outputs/bundle/release/app-release.aab
```

### 4-4. Google Play Console 前提

Play Console で次を満たしてから AAB をアップロードします。

- デベロッパーアカウント確認が完了し、`アプリを作成` が有効であること
- `com.daidoko.app` の新規アプリを作成できること
- Play アプリ署名を有効化し、アップロードキーと一致する AAB を使うこと
- 最初の配布先は内部テストを優先し、警告・審査項目を確認してから公開へ進むこと

Play Console に `新しいアプリを作成するには、アカウント確認を完了してください` と表示される場合、そのアカウントでは作成・アップロード・公開を続行できません。アカウント所有者が確認フローを完了し、Google 側の処理が終わってから再開します。

## 5. 端末へインストール

### 5-1. 接続確認

```powershell
cd C:\Projects\daidoko

& $ADB devices
```

実機 Pixel 9a の検証済みシリアル:

```text
5B131JEBF16695
```

以降の例では `$DEVICE` に対象端末を指定します。

```powershell
$DEVICE = "5B131JEBF16695"
$APK = "apps/mobile/android/app/build/outputs/apk/release/app-release.apk"
```

### 5-2. データを残して更新

```powershell
& $ADB -s $DEVICE install -r $APK
```

同じ versionCode の APK を入れ直すローカル検証で downgrade 扱いになる場合のみ、次を使います。

```powershell
& $ADB -s $DEVICE install -r -d $APK
```

`INSTALL_FAILED_UPDATE_INCOMPATIBLE` が出た場合は署名が変わっています。データ保持が必要な端末ではアプリ削除で解決せず、同じ署名の APK を使うか、アプリ内バックアップを作成してから移行手順を決めます。

### 5-3. バージョン確認

```powershell
& $ADB -s $DEVICE shell dumpsys package com.daidoko.app |
  Select-String "versionName|versionCode"
```

期待値:

```text
versionCode=10003
versionName=1.0.1
```

## 6. E2E 検証

E2E は release APK をインストールした端末に対して実行します。端末はロック解除済みにし、画面を点灯した状態にします。

### 6-1. 実機 Pixel 9a

```powershell
cd C:\Projects\daidoko

& $ADB -s 5B131JEBF16695 shell cmd statusbar collapse
& $ADB -s 5B131JEBF16695 shell input keyevent KEYCODE_WAKEUP
& $ADB -s 5B131JEBF16695 shell wm dismiss-keyguard

$env:TARGET_DEVICE = "5B131JEBF16695"
$env:ADB_PATH = $ADB
node e2e/android-e2e.mjs
```

v1.0.1 の検証結果:

```text
結果: PASS 13 / FAIL 0 / 合計 13
```

検証範囲:

- アプリ起動とホーム描画
- フィルタータブと主要タブ遷移
- 手動レシピ作成、一覧、詳細、削除
- 料理中モードと調理記録写真 UI
- タイムライン表示
- URL 取り込みの応答
- テキスト取り込み AI 指示コピー
- OCR 入口画面
- 設定、通常バックアップ、機種変更バックアップ、ライセンス情報、家族グループ

### 6-2. Emulator

Emulator でも同じ script を使います。`TARGET_DEVICE` は `adb devices` のシリアルへ置き換えます。

```powershell
cd C:\Projects\daidoko

$env:TARGET_DEVICE = "emulator-5554"
$env:ADB_PATH = $ADB
node e2e/android-e2e.mjs
```

スクリーンショットと UI dump は以下に保存されます。

```text
e2e/screenshots/e2e-android
e2e/ui-dumps
```

## 7. 手動スモークチェック

E2E 通過後、必要に応じて次の手動確認を行います。

- 設定 → バックアップ・復元 → バックアップを作成
- 設定 → バックアップ・復元 → 最新バックアップから復元
- 設定 → バックアップ・復元 → 移行ファイルを作成
- 設定 → バックアップ・復元 → 最新移行ファイルを共有
- 設定 → バックアップ・復元 → 移行ファイルから復元
- 設定 → ライセンス情報 → OSS ライセンス一覧表示
- レシピ詳細 → 調理開始 → 完成して記録 → 写真追加 UI 表示
- アプリ再起動後、既存レシピと調理ログが残っていること

## 8. GitHub 反映とタグ

すべてのチェックが通ったら変更をコミットし、`develop` を push します。

```powershell
cd C:\Projects\daidoko

git status --short
git add <変更ファイル>
git commit -m "..."
git push origin develop
```

v1.0.1 タグを新規作成する場合:

```powershell
git tag -a v1.0.1 -m "v1.0.1"
git push origin v1.0.1
```

Google Play 提出アーティファクトに紐づくタグを残す場合:

```powershell
git tag -a v1.0.1-play-10003 378e6f0 -m "Google Play production bundle 10003 (v1.0.1)"
git push origin v1.0.1-play-10003
```

今回の production 審査送信記録:

- 送信日: 2026-06-04
- Git commit: `378e6f0` (`fix: patch android 15 release compatibility`)
- Git tag: `v1.0.1-play-10003`
- versionName / versionCode: `1.0.1` / `10003`
- Play 状態: production track を審査送信済み（承認後は自動公開）

既に GitHub 上に存在するタグは、明示的な合意なしに移動・削除・再作成しません。タグ作成後にドキュメントやテスト安定化のコミットを追加した場合は、タグが指すリリース本体のコミットと、追加コミットの目的をリリースメモまたは PR に明記します。

## 9. v1.0.1 では不要なデプロイ

次の作業は v1.0.1 の必須デプロイ手順ではありません。

- Railway サーバーの新規作成・更新
- PostgreSQL などの外部 DB 作成
- EAS Build のクラウドビルド
- TestFlight / App Store Connect 提出
- クラウド同期用の認証・API 設定

将来サーバー機能や iOS 配布を有効化する場合は、この手順書とは別に、データ移行、バックアップ、ロールバック、署名、ストア審査を含むデプロイ手順を作成します。

## 10. トラブルシューティング

| 症状                                            | 対処                                                                                                                                |
| ----------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `No authorized device`                          | 端末の USB デバッグ許可を確認し、`adb devices` で `device` 表示になるまで待つ。                                                     |
| `uiautomator dump` が固まる                     | Node E2E 本体は止めず、固まった `adb ... shell uiautomator dump /sdcard/_e2e.xml` の子プロセスだけ停止する。                        |
| 通知シェードやクイック設定が UI dump に混入する | `adb shell cmd statusbar collapse` を実行する。E2E script も通知シェード検出時に collapse して再試行する。                          |
| `INSTALL_FAILED_UPDATE_INCOMPATIBLE`            | 署名不一致。データ保持が必要なら uninstall せず、同じ署名の APK で更新する。                                                        |
| `bundleRelease requires DAIDOKO_UPLOAD...`      | Play 用 AAB に必要なアップロード署名情報が未設定。環境変数または `apps/mobile/android/keystore.properties` を設定する。             |
| `アプリを作成` が無効                           | Play Console のデベロッパーアカウント確認が未完了。確認フロー完了と Google 側の処理完了を待ってから再開する。                       |
| versionName / versionCode が違う                | `apps/mobile/app.json` を確認し、`pnpm --filter mobile exec expo prebuild --platform android --no-install` 後に再ビルドする。       |
| Play review で target API 34 エラーが出る       | `pnpm install` で patch を適用し、`pnpm --filter mobile exec expo prebuild --platform android --no-install` 後に AAB を再生成する。 |
| Gradle の `.cxx` 周辺でファイルロックする       | `apps/mobile/android/gradlew.bat --stop` 後、関連 Java プロセスを停止してから再ビルドする。                                         |
| 既存データが見えない                            | アプリ削除、`pm clear`、別 package、署名違いによる再インストールが起きていないか確認する。                                          |
