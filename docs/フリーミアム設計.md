# フリーミアム設計 — AI 写真レシピ

> 「写真からレシピ」（Vision 推論）を**フリーミアム機能**として提供するための設計。
> 実装前・課金まわりの変更前に必ず参照すること。関連: `docs/レシピ作成フロー.md` / `docs/アーキテクチャ設計.md` / `infra/README.md`

---

## 1. なぜフリーミアムか

「写真からレシピ」は唯一**変動コスト**が発生する機能（Gemini Vision API は呼び出し従量課金）。
無制限に無料提供するとコストが青天井になるため、

- **無料枠**: 1 日 1 回まで（お試し＋日常のライトユース）
- **プレミアム**: 月額サブスクで無制限

とする。コスト源は AI 呼び出しだけなので、**課金対象は AI 写真レシピのみ**。
手動入力・URL 取り込み・OCR・調理記録・家族共有などは**すべて無料・無制限**のまま。

---

## 2. プラン定義

| 項目                     | 無料（Free）  | プレミアム（Premium）           |
| ------------------------ | ------------- | ------------------------------- |
| AI 写真レシピ            | **1 日 1 回** | **無制限**                      |
| 手動入力 / URL / OCR     | 無制限        | 無制限                          |
| 調理記録・家族共有・検索 | 無制限        | 無制限                          |
| 課金                     | なし          | **月額サブスク**（例: ¥300/月） |

- 無料枠は**暦日（YYYY-MM-DD）単位**でリセット。
- 無料ユーザーは**リワード広告 1 本で当日 +1 回**（最大 3 回/日）。詳細は §7。
- 価格はストア側（Play / App Store）で設定する（コード非依存）。

---

## 3. アーキテクチャ方針

本アプリは**アカウント／認証を持たない**（ローカル SQLite のみ）。
そのため freemium は次の二層で構成する。

```
┌──────────────────────── 端末（クライアント）────────────────────────┐
│  エンタイトルメント  : RevenueCat（端末匿名 App User ID／レシート検証）│
│  無料枠カウント      : SQLite app_meta（端末ローカル・日次）          │
│  ゲート              : import-photo 画面で canInfer を判定           │
└────────────────────────────────────────────────────────────────────┘
                                  │ 課金状態の真実 = ストアのレシート
                                  ▼
┌──────────────────────── サーバー（コスト天井）──────────────────────┐
│  グローバル日次上限  : INFER_GLOBAL_DAILY_LIMIT（全体のコスト上限）  │
│  Gemini クォータ     : Google AI Studio 側で requests/day を上限化   │
└────────────────────────────────────────────────────────────────────┘
```

### なぜクライアント側でエンタイトルメントを判定するのか

- アカウントが無いため、サーバーで「誰が何回使ったか」を厳密に紐付けられない。
- **プレミアム判定は RevenueCat**（＝ Google/Apple のレシート検証）に委ねるため、
  端末ローカルでも**なりすましは困難**（無料ユーザーが prem を詐称できない）。
- **無料枠カウントは端末ローカル**＝再インストールでリセットされる「ソフト」な上限。
  これは許容する。理由は、**全体のコスト上限はサーバーの global cap が担保**するため。
  「無料枠の厳密な個人単位カウント」が必要になったら、アカウント＋サーバー集計（DynamoDB 等）へ拡張する（将来）。

## 4. データモデル（端末ローカル）

新規テーブルは作らず、既存の `app_meta`（KV）を再利用する。

| key                                | value      | 意味                           |
| ---------------------------------- | ---------- | ------------------------------ |
| `ai_photo_recipe_usage:YYYY-MM-DD` | 整数文字列 | その日の AI 写真レシピ成功回数 |

- 日が変わると参照キーが変わる ⇒ 実質的に**自動リセット**。
- 旧日のキーは残るが微小（1 日 1 行）。

## 5. 判定ロジック

```
FreemiumStatus = {
  isPremium,                       // RevenueCat entitlements.active['premium']
  used,                            // 当日の成功回数
  limit = 1,                       // FREE_DAILY_LIMIT
  remaining = max(0, limit - used),// premium は Infinity
  canInfer = isPremium || remaining > 0,
}
```

### フロー（import-photo 画面）

1. 画面表示時に `getFreemiumStatus()` を取得し、
   - プレミアム → 「プレミアム（無制限）」
   - 無料 → 「今日あと N 回」
     を表示。
2. 「撮影」「ギャラリーから選ぶ」タップ時、`canInfer === false` なら**ペイウォール**へ遷移して中断。
3. **クラウド推論が成功**したときだけ `recordCloudInference()` を呼び、無料ユーザーのカウントを +1。
   - **料理でない（not_a_dish）** や **AI に繋がらず端末内フォールバック**したケースは**カウントしない**
     （ユーザーは AI 結果を受け取っていない／コストも実質発生しないため、ユーザーフレンドリーに）。
4. サーバーは引き続き global cap で全体コストを抑える（多重防御）。

> **写真送信の同意**: 写真は解析のためサーバー（AI 提供元）へ送信される。ブロッキングの同意ダイアログは廃止し、
> 画面内に常時その旨を明示する方式に変更（毎回確認しない）。プライバシーポリシー／ストアの Data safety で開示する。

## 6. 課金（RevenueCat）

- ライブラリ: `react-native-purchases`（Expo dev/release ビルドで autolink。設定プラグイン不要）。
- エンタイトルメント識別子: **`premium`**。
- 月額サブスクの **Offering**（current）から `monthly` パッケージを取得して購入。
- 復元: `restorePurchases()`（ストアアカウントに購入が紐づく）。

### プロバイダ抽象（差し替え可能）

```
EntitlementProvider {
  configure(): Promise<void>
  isPremium(): Promise<boolean>
  getOffering(): Promise<PremiumOffering | null>   // 価格表示用
  purchasePremium(): Promise<{ success: boolean }> // ユーザーキャンセルは success:false
  restore(): Promise<boolean>
}
```

- `RevenueCatEntitlementProvider`: `EXPO_PUBLIC_REVENUECAT_API_KEY` がある & ネイティブ環境のときに使用。
- `StubEntitlementProvider`: キー未設定 / Web / テスト時のフォールバック。
  `isPremium=false`、購入系は「準備中」エラー（＝アプリは**無料枠だけで完全に動く**）。

## 7. リワード広告（広告を見て 1 回ぶん）

無料枠を使い切った無料ユーザーは、**リワード広告を 1 本見ると当日 +1 回**使える（1 日 **3 回**まで＝最大 1+3=4 回/日）。
プレミアムは広告なし・無制限。バナー／インタースティシャルは世界観に合わないため**使わない**
（opt-in のリワード ＋ 強ガード付きのアプリ起動広告のみ — §7-2、2026-07-12 決定）。

- 算定: `usage.service.ts`（`AD_BONUS_DAILY_LIMIT=3`、`app_meta` キー `ai_photo_recipe_ad_bonus:YYYY-MM-DD`、
  `grantAdBonus()`、`getFreemiumStatus().canWatchAdForMore`）。
- 抽象: `AdRewardProvider`（`ad-reward.types.ts`）— `isAvailable()` / `showRewardedAd(): Promise<{ rewarded }>`。
- プロバイダ: `ad-reward.service.ts` の factory が `EXPO_PUBLIC_ADMOB_ENABLED=true` & ネイティブ時に
  `AdMobRewardProvider`（`ad-reward.admob.ts`／web は `.web` スタブ）を返す。**既定 false ＝ 広告 UI 非表示**で挙動不変。
- 導線: ペイウォール（`paywall.tsx`）に「広告を見て 1 回ぶん使う」ボタン（`canWatchAdForMore` のとき表示）→ 完了で `grantAdBonus()`。

### AdMob の状態と本番化（配線済み）

`react-native-google-mobile-ads` は**導入・配線済み**。`app.json` の設定プラグインには Google の
**テスト アプリ ID** を入れてある（未設定だと起動クラッシュするため必須）。実装は `ad-reward.admob.ts`。

- **開発で動かす**: `EXPO_PUBLIC_ADMOB_ENABLED=true` でビルド → ペイウォールの「広告を見て…」で Google の
  **テスト広告**が出る（`EXPO_PUBLIC_ADMOB_REWARDED_UNIT_ID` 未設定なら `TestIds.REWARDED`）。
  app.json のプラグイン変更を反映する場合はローカルビルドに `--prebuild` が必要。
- **同意管理（UMP / GDPR）— 実装済み（2026-07-05）**:
  - 広告初期化前に `AdsConsent.gatherConsent()` を実行（`ad-reward.admob.ts`）。GDPR 対象地域では
    同意フォームが表示され、日本など対象外地域ではフォームなしで進む。`canRequestAds` が立たない場合は
    `AdUnavailableError` で広告ボタンのエラーメッセージに落ちる。
  - パーソナライズは UMP/TCF の同意状態に SDK が従う（`requestNonPersonalizedAdsOnly` の固定は廃止）。
  - 設定 → プランに「広告のプライバシー設定」行（`privacyOptionsRequirementStatus === REQUIRED` の
    ユーザーのみ表示）→ `showPrivacyOptionsForm()` で同意をいつでも変更可能。
  - `app.json` プラグインに `delayAppMeasurementInit: true`（同意前の SDK 計測開始を防ぐ）と
    iOS ATT 用 `userTrackingUsageDescription` を設定済み。
- **本番化（あなたの作業）**: `docs/リリース手順.md` §6 参照（AdMob 本番 ID・AD_ID 権限の解除・
  Play 広告申告/データセーフティ・課金サンドボックス検証）。
  ネイティブのリンク確認はこのビルドが初回（JS の typecheck/lint/test は緑）。

> リワードで増えた利用もサーバーを叩く。広告で件数が伸びるなら `INFER_GLOBAL_DAILY_LIMIT` を引き上げる（広告が原資）。
> 採算目安: Gemini(flash) 1 回 ≈ 0.1 円未満 ＜ リワード広告 1 本 ≈ 0.7〜2 円。

### 7-2. アプリ起動広告（App Open — 2026-07-12 追加）

リワードだけでは表示機会が「無料枠を使い切った人」に限られ収益母数が小さいため、
**フォアグラウンド復帰時のアプリ起動広告**を 1 枠だけ追加（ユーザー要望「使い勝手を著しく損なわない範囲で最適化」）。

- 実装: `app-open-ad.service.ts`（判定の純関数 `evaluateAppOpenAdGate` ＋ 配線は `app/_layout.tsx` の AppState リスナー）。
  プロバイダは `app-open-ad.admob.ts`（/`.web` スタブ）。ユニット ID は `EXPO_PUBLIC_ADMOB_APP_OPEN_UNIT_ID`（未設定 = テスト ID）。
- **表示ガード（全部 AND・どれか欠けたら出さない）**:
  1. Android の広告有効ビルド、かつプレミアムでない
  2. 写真の撮影/ギャラリー往復からの復帰ではない（`photo-capture.service` がフラグ連動）
  3. タイマーが `idle`（調理中は出さない）かつ調理・カメラ系画面（/cook, /receipt, /import-photo, /import-ocr, /consume-meal, /scan-barcode）にいない
  4. **60 秒以上**アプリを離れていた（アプリ切替の往復では出さない）
  5. 初回起動から **24 時間**の猶予期間を過ぎている
  6. 前回表示から **6 時間**以上（≒ 1 日最大 2〜3 回が上限）
- コールドスタートでは出さない（起動を遅らせない）。表示状態は `app_meta`
  （`app_open_ad_last_shown_at` / `app_open_ad_first_eligible_at`）に永続化。
- 失敗は全て握りつぶすベストエフォート — 広告都合でアプリの動作を妨げない。
- **ユニット ID 未設定なら完全オフ**（本番にテストユニットが紛れる事故防止）。実機検証は
  `EXPO_PUBLIC_ADMOB_APP_OPEN_UNIT_ID` に Google のテスト ID を明示指定してビルドする。
  本番有効化 = AdMob でアプリ起動広告ユニットを作成 → eas.json の production env に ID を追加（1.5.0 以降）。

## 8. 残る境界（あなたの作業）

コード（無料枠・ペイウォール・課金抽象・リワード広告・**UMP 同意管理**・BYOK）は実装済み。
以下は**ストア側のアカウント・商品設定が必要なため必ず手元に残る**作業。
手順の詳細と申告変更（AD_ID）は `docs/リリース手順.md` §6。

1. **RevenueCat** アカウント作成 → プロジェクト → Android の public SDK key 取得。
2. RevenueCat で **エンタイトルメント `premium`** と、月額サブスクを含む **Offering** を作成。
3. **Google Play Console** で支払いプロファイルを設定し、月額サブスク商品を作成して RevenueCat に紐付け。
4. **AdMob** でアプリ＋リワード広告ユニットを作成。
5. ID が揃ったら eas.json / app.json に設定注入（リリース手順 §6-2）→ ライセンステスターで
   **実購入フローとリワード広告を検証**（§6-4）。
   - ※ `react-native-purchases` のネイティブリンク確認はこのビルドが初回。JS レベル（typecheck/lint/test）は緑。

## 9. BYOK（自分の Gemini キーを持ち込む）

ユーザーが**自分の Gemini API キー**を設定すると、アプリは**端末から Gemini を直接呼び出す**（サーバー不要）。
このとき**無料枠・課金・広告はすべてバイパス**（無制限）。ユーザーは Google に直接課金されるため、こちらにコストは発生しない。

- なぜサーバー不要か: 自分のキーをアプリに**埋め込むと抽出・悪用される**ためサーバーが要る（managed）。BYOK は
  各ユーザーが**自分のキー・自分の端末・自分の課金**で動かすので共有秘密が無く、サーバーが要らない。
- 保存: `byok.service.ts` が **expo-secure-store**（OS キーチェーン／キーストア、暗号化）に保存。`getUserApiKey()` /
  `setUserApiKey()` / `clearUserApiKey()`。Google 以外には送信しない。
- 直接呼び出し: `vision-recipe.provider.ts` の `inferRecipeFromVision` が、キーがあれば `inferViaByok`（Gemini 直叩き＋
  `normalizeGeminiRaw`）、無ければ `inferViaServer`（managed）に分岐。
- 無制限化: `usage.service.ts` が `hasUserApiKey()` を見て `isByok` を立て、`canInfer=true`／カウントしない。
- 設定 UI: `app/(tabs)/ai-key.tsx`（設定 → プラン → 自分のAIキー）。キー貼り付け・削除・取得方法リンク。
  管理方法・削除方法・制約（機種変更で引き継がれない・Gemini専用）は `InfoTooltip`（タップ展開式の補足行）で明記（2026-07 追記）。
- モデル: `EXPO_PUBLIC_GEMINI_MODEL`（既定 `gemini-2.5-flash`）。
- 前提（あなたの作業）: ユーザー各自が Google AI Studio でキーを取得（アプリ内にリンクあり）。アプリ側の実装は完了。

> **方針メモ**: BYOK と「サブスク／広告で稼ぐ managed」は方向が逆。サーバー運用を完全にやめたいなら
> **純 BYOK**（managed・課金・広告を撤去）も選べる。両立させる場合は本実装のまま（managed 既定＋上級者は BYOK）。
