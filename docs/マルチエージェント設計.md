## 1. 概要

だいどこのバックエンド・モバイル処理は、単一の巨大なサービスではなく、**明確な責務境界を持つ複数のエージェント**として設計する。各エージェントは独立したインターフェース（入力・出力・エラー型）を持ち、疎結合に協調する。

## 3. 共通型定義

```typescript
// packages/shared/types/agent.ts

/** 全エージェントの出力共通型 */
interface AgentResult<T> {
  ok: boolean;
  data?: T;
  error?: AgentError;
}

interface AgentError {
  code: AgentErrorCode;
  message: string;
  retryable: boolean;
}

type AgentErrorCode =
  | 'UNSUPPORTED_SITE' // A4: JSON-LD 非対応サイト
  | 'FETCH_FAILED' // A4: 外部サイト取得失敗
  | 'PARSE_FAILED' // A4/A5: 解析失敗
  | 'AI_API_UNAVAILABLE' // A5: AI API 障害
  | 'OCR_FAILED' // A2: ML Kit エラー
  | 'SYNC_CONFLICT' // A6: 競合解決不能
  | 'NETWORK_UNAVAILABLE' // A3: オフライン
  | 'RATE_LIMITED' // A4/A5/A7: レートリミット超過
  | 'UNAUTHORIZED' // A7: JWT 無効・期限切れ
  | 'FORBIDDEN' // A7: 権限不足
  | 'TOKEN_EXPIRED' // A7: アクセストークン期限切れ（リフレッシュ促進）
  | 'MIGRATION_FAILED' // A8: DB マイグレーション失敗
  | 'DB_INTEGRITY_ERROR' // A8: 整合性チェック失敗
  | 'FTS_REBUILD_FAILED' // A8: FTS5 インデックス再構築失敗
  | 'UNKNOWN';

/** レシピ下書き（全エージェントが生成する共通スキーマ） */
interface RecipeDraft {
  title: string;
  titleReading?: string;
  description?: string;
  servings?: number;
  cookTimeMin?: number;
  prepTimeMin?: number;
  ingredients: IngredientDraft[];
  steps: StepDraft[];
  sourceUrl?: string;
  confidence: 'high' | 'medium' | 'low'; // ユーザーへの確認促進に使用
}

interface IngredientDraft {
  groupLabel?: string;
  name: string;
  amount?: string;
  note?: string;
}

interface StepDraft {
  body: string;
  timerSec?: number;
}
```

## 4. A1 — ImportAgent（URL 取り込み）

### 4.1 責務

クライアント側のオーケストレーターとして、ユーザーから URL を受け取り、サーバーの UrlFetchAgent を呼び出して `RecipeDraft` を返す。ローディング状態・エラー表示・リトライ判定も担う。

### 4.2 インターフェース

```typescript
// apps/mobile/src/agents/importAgent.ts
interface ImportAgentInput {
  url: string;
  signal?: AbortSignal; // タイムアウト・キャンセル用
}

type ImportAgentOutput = AgentResult<RecipeDraft>;

async function runImportAgent(input: ImportAgentInput): Promise<ImportAgentOutput>;
```

### 4.3 ステートマシン

```
Idle
 │── url 入力
 ▼
Validating（URL 形式チェック）
 │── 不正 URL → エラー表示（Idle に戻る）
 │── OK
 ▼
Fetching（POST /api/v1/import/url）
 │── タイムアウト（10秒） → FETCH_FAILED（retryable: true）
 │── UNSUPPORTED_SITE → エラー表示「対応サイトではありません」→ Idle
 │── 成功
 ▼
Previewing（RecipeDraft をユーザーに確認させる）
 │── 編集して保存 → saveRecipe()
 │── キャンセル → Idle
```

### 4.4 エラーハンドリング

| エラーコード          | ユーザー表示                                                        | リトライ        |
| --------------------- | ------------------------------------------------------------------- | --------------- |
| `UNSUPPORTED_SITE`    | 「このサイトには対応していません。手動入力かOCRをご利用ください。」 | ❌              |
| `FETCH_FAILED`        | 「読み込みに失敗しました。もう一度お試しください。」                | ✅（最大 2 回） |
| `RATE_LIMITED`        | 「しばらく待ってからお試しください。」                              | ✅（30 秒後）   |
| `NETWORK_UNAVAILABLE` | 「インターネット接続を確認してください。」                          | ✅（復帰後）    |

## 5. A2 — OCRAgent（カメラ OCR）

### 5.1 責務

端末カメラで撮影した画像を ML Kit でテキスト認識し、ルールベースパーサー（v1.0）または AIStructureAgent（v1.1+）で `RecipeDraft` に変換する。処理は全て端末内で完結する（v1.0）。

### 5.2 インターフェース

```typescript
// apps/mobile/src/agents/ocrAgent.ts
interface OCRAgentInput {
  imageUri: string; // 端末内パス（カメラ撮影済み画像）
  useServerAI?: boolean; // v1.1 以降：サーバー AI 構造化を有効にするか
}

type OCRAgentOutput = AgentResult<RecipeDraft>;

async function runOCRAgent(input: OCRAgentInput): Promise<OCRAgentOutput>;
```

### 5.3 処理フロー

```
撮影画像（imageUri）
 │
 ▼
[A2-1] 前処理（expo-image-manipulator）
  ・1200px にリサイズ、グレースケール変換
  ・ファイルサイズ ≤ 2MB に圧縮
 │
 ▼
[A2-2] 文字認識（ML Kit Text Recognition / Apple Vision）
  ・出力: 認識テキスト文字列（改行・ブロック情報付き）
  ・失敗 → OCR_FAILED
 │
 ▼
[A2-3a] ルールベースパーサー（v1.0, 常時実行）
  ・STEP_PATTERN / AMOUNT_PATTERN / SECTION_PATTERN で解析
  ・confidence: 'medium' を付与
 │
 ├── useServerAI=true かつ confidence='low' の場合
 │    ▼
 │   [A2-3b] AIStructureAgent 呼び出し（POST /api/v1/ai/structure）
 │    ・テキストのみ送信（画像は送らない）
 │    ・成功 → confidence: 'high' に更新
 │    ・失敗 → A2-3a の結果をそのまま使用
 │
 ▼
RecipeDraft（Previewing 画面へ）
```

### 5.4 プライバシー制約

- ML Kit による文字認識処理は**全て端末内で実行**。画像データはサーバーへ送信しない。
- AIStructureAgent へ送信するのは**認識済みテキストのみ**。写真・位置情報・個人情報は含めない。
- ユーザーが v1.1 のサーバー AI 構造化を有効にする場合、初回のみ同意確認を表示する。

## 6. A3 — SyncAgent（バックグラウンド同期）

### 6.1 責務

`SyncQueue` テーブルに積まれたローカル変更をサーバーへ送信し、サーバーからの差分をローカルに反映する。ネットワーク状態を監視してバックグラウンドで自動実行する。

### 6.2 インターフェース

```typescript
// apps/mobile/src/agents/syncAgent.ts
interface SyncAgentConfig {
  maxRetry: number; // デフォルト: 3
  batchSize: number; // 一度に送信するキュー件数。デフォルト: 50
  flushIntervalMs: number; // 定期フラッシュ間隔。デフォルト: 30000
}

interface SyncResult {
  pushed: number; // サーバーへ送信した件数
  pulled: number; // サーバーから受信した件数
  conflicts: number; // 競合が発生した件数
  failed: number; // 失敗した件数
}

async function flushSyncQueue(): Promise<SyncResult>;
```

### 6.3 起動トリガー

| トリガー                   | タイミング                               |
| -------------------------- | ---------------------------------------- |
| アプリフォアグラウンド復帰 | `AppState` 変化で検知                    |
| ネットワーク接続回復       | `@react-native-community/netinfo` で検知 |
| 定期フラッシュ             | 30 秒ごと（フォアグラウンド時のみ）      |
| `saveRecipe()` 完了直後    | 手動呼び出し                             |

### 6.4 エンティティ別同期戦略

| エンティティ           | 方向      | 競合解決                        | 備考             |
| ---------------------- | --------- | ------------------------------- | ---------------- |
| Recipe                 | 双方向    | last-write-wins（updatedAt）    |                  |
| RecipeRevision         | Push のみ | 競合なし（immutable INSERT）    |                  |
| Ingredient / Step      | Push のみ | RecipeRevision に従属           |                  |
| CookingLog / Photo     | Push のみ | 競合なし（追記）                |                  |
| Memo                   | 双方向    | Vector Clock → 競合時マージ画面 |                  |
| ShoppingItem.isChecked | 双方向    | last-write-wins                 | リアルタイム優先 |

### 6.5 失敗・リトライ戦略

```
retryCount < maxRetry(3) → exponential backoff（1s, 4s, 16s）
retryCount = maxRetry     → エラー通知（通知バッジ）。手動リトライを促す
```

## 7. A4 — UrlFetchAgent（サーバー側 URL 解析）

### 7.1 責務

クライアントから URL を受け取り、外部サイトをサーバーサイドでフェッチし、JSON-LD Schema.org Recipe を抽出・正規化して `RecipeDraft` を返す。

### 7.2 インターフェース

```typescript
// apps/server/src/agents/urlFetchAgent.ts
interface UrlFetchInput {
  url: string;
  requesterId: string; // レートリミット管理用ユーザーID
}

type UrlFetchOutput = AgentResult<RecipeDraft>;

async function runUrlFetchAgent(input: UrlFetchInput): Promise<UrlFetchOutput>;
```

### 7.3 処理詳細

```typescript
async function runUrlFetchAgent(input: UrlFetchInput): Promise<UrlFetchOutput> {
  // 1. URL バリデーション（https のみ許可）
  const parsedUrl = new URL(input.url);
  if (parsedUrl.protocol !== 'https:') {
    return error('FETCH_FAILED', '対応プロトコルは HTTPS のみです', false);
  }

  // 2. レートリミット確認（10 req/min/user）
  const limited = await rateLimit.check(input.requesterId, 10, 60);
  if (limited) return error('RATE_LIMITED', '...', true);

  // 3. 外部フェッチ（タイムアウト 8 秒）
  const html = await fetch(parsedUrl, {
    signal: AbortSignal.timeout(8000),
    headers: { 'User-Agent': 'DaidokoBot/1.0' },
  }).then((r) => r.text());

  // 4. JSON-LD 抽出
  const jsonLd = extractJsonLd(html); // <script type="application/ld+json"> を全て取得
  const recipeSchema = jsonLd.find((j) => j['@type'] === 'Recipe');
  if (!recipeSchema) {
    return error('UNSUPPORTED_SITE', 'Recipe schema が見つかりません', false);
  }

  // 5. 正規化
  return { ok: true, data: normalizeRecipeSchema(recipeSchema) };
}
```

### 7.4 正規化ルール（`normalizeRecipeSchema`）

| フィールド               | Schema.org → RecipeDraft | 変換処理                                                 |
| ------------------------ | ------------------------ | -------------------------------------------------------- |
| `name`                   | `title`                  | そのまま                                                 |
| `recipeIngredient[]`     | `ingredients[]`          | UNIT_PATTERN regex で `name`/`amount` に分割             |
| `recipeInstructions`     | `steps[]`                | string[] / HowToStep[] / HowToSection[] を再帰的に平坦化 |
| `recipeYield`            | `servings`               | 数字部分を抽出（"4人分" → 4）                            |
| `totalTime` / `cookTime` | `cookTimeMin`            | ISO 8601 duration を分に変換（"PT30M" → 30）             |
| `description`            | `description`            | そのまま                                                 |

### 7.5 対応サイト

| サイト             | 対応状況         | JSON-LD 提供             |
| ------------------ | ---------------- | ------------------------ |
| クラシル           | ✅               | あり                     |
| デリッシュキッチン | ✅               | あり                     |
| Nadia              | ✅               | あり                     |
| NHKきょうの料理    | ❌（スコープ外） | なし                     |
| クックパッド       | ❌（スコープ外） | bot 検知・JSON-LD 非対応 |
| 個人ブログ等       | ❌               | 不定 → UNSUPPORTED_SITE  |

### 8.3 プロンプト設計

```typescript

以下のルールを厳守してください:
```

### 8.4 フォールバック戦略

```
AI API 呼び出し
 │
 ├── 成功（レスポンス JSON が RecipeDraft 型に合致）
 │    → confidence: 'high' で返却
 │
 ├── 成功（JSON は返ってきたが型不合致）
 │    → Zod で部分的に解析し、合致したフィールドのみ採用
 │    → confidence: 'medium' で返却
 │
 └── 失敗（タイムアウト / API エラー / JSON パース失敗）
      → ルールベースパーサーの結果を confidence: 'low' で返却
      → A5 のエラーはクライアントには露出しない
```

### 8.5 プライバシー・コスト管理

| 項目                    | 方針                                                                        |
| ----------------------- | --------------------------------------------------------------------------- |
| 送信データ              | レシピテキストのみ。写真・ユーザー情報・家族情報は一切送らない              |
| ログ保持                | AI API へのリクエスト内容はサーバーログに残さない（プライバシー保護）       |
| コスト上限              | requesterId ごとに 5 req/min のレートリミット。月次コスト監視アラートを設定 |
| AI プロバイダー選定基準 | Zero Data Retention（ZDR）ポリシーを有するプロバイダーを優先選定            |

## 9. A6 — SyncProcessorAgent（サーバー側同期処理）

### 9.1 責務

`POST /api/v1/sync/push` を受け取り、エンティティ種別ごとに適切な DB 操作を実行する。競合検出・解決ロジックを担う。

### 9.2 インターフェース

```typescript
// apps/server/src/agents/syncProcessorAgent.ts
interface SyncPushInput {
  userId: string;
  familyId: string;
  items: SyncQueueItem[];
}

interface SyncPushOutput {
  accepted: string[]; // 受理した entityId 一覧
  rejected: RejectedItem[];
}

interface RejectedItem {
  entityId: string;
  reason: 'CONFLICT' | 'PERMISSION_DENIED' | 'SCHEMA_MISMATCH';
  serverVersion?: unknown; // CONFLICT 時にクライアントへ返す
}
```

### 9.3 エンティティ別処理ロジック

```typescript
async function processItem(item: SyncQueueItem, ctx: SyncContext) {
  switch (item.entityType) {
    case 'Recipe':
      // last-write-wins: サーバーの updatedAt と比較
      await db.recipe.upsert({
        where: { id: item.entityId },
        create: item.payload,
        update: item.updatedAt > serverUpdatedAt ? item.payload : noChange,
      });
      break;

    case 'RecipeRevision':
    case 'CookingLog':
    case 'CookingPhoto':
      // immutable INSERT: 重複は UNIQUE 制約で無視
      await db.execute(`INSERT OR IGNORE INTO ${item.entityType} VALUES (...)`, item.payload);
      break;

    case 'Memo':
      // Vector Clock 比較
      const serverClock = await db.memo.getClock(item.entityId);
      const result = compareVectorClocks(item.vectorClock, serverClock);
      if (result === 'concurrent') {
        rejected.push({ entityId: item.entityId, reason: 'CONFLICT', serverVersion });
      } else {
        await db.memo.upsert(item.payload);
      }
      break;

    case 'ShoppingItem':
      // isChecked のみ last-write-wins
      await db.shoppingItem.update({
        where: { id: item.entityId },
        data: { isChecked: item.payload.isChecked, updatedAt: item.updatedAt },
      });
      break;
  }
}
```

### 9.4 権限チェック

| エンティティ                       | Owner | Member |
| ---------------------------------- | ----- | ------ |
| Recipe（作成・編集・削除）         | ✅    | ❌     |
| Recipe（閲覧）                     | ✅    | ✅     |
| RecipeRevision（派生版作成）       | ✅    | ✅     |
| CookingLog（自分の記録）           | ✅    | ✅     |
| CookingLog（他メンバーの記録編集） | ✅    | ❌     |
| ShoppingList（共有）               | ✅    | ✅     |
| Memo（自分の非公開メモ）           | ✅    | ✅     |

## 10. A7 — SecurityAgent（セキュリティ専門エージェント）

### 10.1 責務

全サーバーリクエストを横断するミドルウェアとして機能し、JWT 検証・認可・レートリミット・入力サニタイズ・監査ログを一元管理する。他のエージェント（A4/A5/A6）は SecurityAgent が通過を許可したリクエストのみを処理する。

```
HTTP Request
    │
    ▼
┌──────────────────────────────────────────────┐
│              SecurityAgent (A7)              │
│                                              │
│  1. JWT 検証        → UNAUTHORIZED / TOKEN_EXPIRED
│  2. レートリミット  → RATE_LIMITED
│  3. 入力サニタイズ  → PARSE_FAILED
│  4. 認可チェック    → FORBIDDEN
│  5. 監査ログ書き込み                          │
│                                              │
└──────────────────┬───────────────────────────┘
                   │ 通過
```

### 10.3 JWT 検証（ステップ1）

```typescript
async function verifyJwt(c: Context): Promise<void> {
  const authHeader = c.req.header('Authorization');
  if (!authHeader?.startsWith('Bearer ')) throw new SecurityError('UNAUTHORIZED');

  const token = authHeader.slice(7);
  try {
    // RS256 で検証（公開鍵はサーバー起動時にロード）
    const payload = await jwtVerify(token, PUBLIC_KEY, { algorithms: ['RS256'] });

    // exp チェック（jose が自動実施するが明示的に確認）
    if (payload.exp && payload.exp < Date.now() / 1000) {
      throw new SecurityError('TOKEN_EXPIRED');
    }

    c.set('secCtx', {
      userId: payload.sub,
      familyId: payload.familyId,
      role: payload.role,
      deviceId: payload.deviceId,
      requestId: crypto.randomUUID(),
    } satisfies SecurityContext);
  } catch (e) {
    if (e instanceof SecurityError) throw e;
    throw new SecurityError('UNAUTHORIZED');
  }
}
```

### 10.5 入力サニタイズ（ステップ3）

```typescript
async function sanitizeInput(c: Context): Promise<void> {
  const body = await c.req.json().catch(() => null);
  if (!body) return;

  // Zod スキーマで型・長さ・文字種をバリデーション
  const schema = getRequestSchema(c.req.path, c.req.method);
  const result = schema.safeParse(body);
  if (!result.success) throw new SecurityError('PARSE_FAILED', result.error.message);

  // パース済みデータを Context に保存（生 body は破棄）
  c.set('validatedBody', result.data);
}
```

バリデーションルール（Zod レベル）:

| フィールド種別               | 制約                                         |
| ---------------------------- | -------------------------------------------- |
| URL 文字列                   | `z.string().url()` + `https:` プロトコルのみ |
| フリーテキスト（タイトル等） | `z.string().max(200).trim()`                 |
| UUID                         | `z.string().uuid()`                          |
| 数値                         | `z.number().int().min(0).max(99999)`         |
| HTML / スクリプト含む文字列  | 拒否（`/<[^>]*>/.test()` でチェック）        |

### 10.6 認可（ステップ4）

```typescript
// エンドポイントごとに必要ロールを宣言的に定義
const ENDPOINT_POLICIES: Record<string, 'any' | 'owner'> = {
  'POST /api/v1/recipes': 'owner',
  'PUT /api/v1/recipes/:id': 'owner',
  'DELETE /api/v1/recipes/:id': 'owner',
  'POST /api/v1/sync/push': 'any', // エンティティ別の細粒度チェックは A6 が担当
  'POST /api/v1/import/url': 'any',
  'POST /api/v1/ai/structure': 'any',
};

async function authorize(c: Context): Promise<void> {
  const policy = ENDPOINT_POLICIES[`${c.req.method} ${c.req.routePath}`];
  const ctx = c.get('secCtx') as SecurityContext;
  if (policy === 'owner' && ctx.role !== 'owner') {
    throw new SecurityError('FORBIDDEN');
  }
}
```

## 11. A8 — DatabaseAgent（データベース専門エージェント）

### 11.1 責務

モバイル（SQLite）とサーバー（PostgreSQL）の双方で、スキーマ移行・FTS5 インデックス同期・クエリ最適化・整合性チェック・定期メンテナンスを担う。アプリロジックからの直接 SQL 実行を仲介し、安全なデータアクセス層を提供する。

### 11.2 実行環境別の役割分担

| 役割             | Mobile (SQLite + Drizzle)          | Server (PostgreSQL + Drizzle)      |
| ---------------- | ---------------------------------- | ---------------------------------- |
| スキーマ移行     | アプリ起動時に自動マイグレーション | デプロイ時に `drizzle-kit migrate` |
| FTS5 / 全文検索  | SQLite FTS5 仮想テーブル管理       | PostgreSQL `pg_trgm` / `tsvector`  |
| 整合性チェック   | `PRAGMA integrity_check`           | `pg_catalog` ヘルスチェック        |
| 定期メンテナンス | `PRAGMA wal_checkpoint` / VACUUM   | VACUUM ANALYZE（週次）             |
| クエリ監視       | slow query ログ（>100ms）          | `pg_stat_statements`               |

### 11.3 インターフェース

### 11.4 Mobile — SQLite 初期化フロー

```typescript
// apps/mobile/src/db/init.ts
export async function initDatabase(): Promise<void> {
  const db = await openDatabaseAsync('daidoko.db');

  // WAL モードと最適化 PRAGMA（アーキテクチャ設計書 §3.3 参照）
  await db.execAsync(`
    PRAGMA journal_mode = WAL;
    PRAGMA foreign_keys = ON;
    PRAGMA cache_size = -8000;   -- 8MB キャッシュ

    PRAGMA mmap_size = 268435456; -- 256MB mmap
```

### 11.5 FTS5 インデックス設計

FTS5 仮想テーブルで `title`・`titleReading`・`ingredientNames`（スペース区切り結合）を全文検索する。

```sql
-- FTS5 仮想テーブル
CREATE VIRTUAL TABLE IF NOT EXISTS recipe_fts USING fts5(
  recipeId UNINDEXED,
  title,
  titleReading,
  ingredientNames,    -- "玉ねぎ にんじん じゃがいも" のように結合して格納
  tokenize = 'unicode61 remove_diacritics 1'
);

-- Recipe 保存後に呼び出す同期処理
INSERT OR REPLACE INTO recipe_fts(recipeId, title, titleReading, ingredientNames)
SELECT
  r.id,
  rv.title,
  rv.titleReading,
  (SELECT GROUP_CONCAT(i.name, ' ')
   FROM Ingredient i
   WHERE i.revisionId = r.currentRevisionId) AS ingredientNames
FROM Recipe r
JOIN RecipeRevision rv ON rv.id = r.currentRevisionId
WHERE r.id = ?;
```

FTS5 同期のタイミング:

| トリガー                           | 呼び出し                          |
| ---------------------------------- | --------------------------------- |
| `saveRecipe()` 完了後              | `DatabaseAgent.syncFts(recipeId)` |
| アプリ起動時（前回クラッシュ検知） | `DatabaseAgent.rebuildFts()`      |
| データインポート完了後             | `DatabaseAgent.rebuildFts()`      |

### 11.6 整合性チェックスケジュール

```typescript
// apps/mobile/src/db/maintenance.ts
async function runScheduledMaintenance(): Promise<void> {
  // アプリ起動から 30 秒後、バックグラウンドで実行
  await sleep(30_000);

  // WAL チェックポイント（未チェックポイントのページを解消）
  await DatabaseAgent.checkpoint();

  // 週次: 整合性チェック（最終実行から 7 日以上経過した場合のみ）
  const lastCheck = await AsyncStorage.getItem('lastIntegrityCheck');
  if (!lastCheck || Date.now() - Number(lastCheck) > 7 * 86400_000) {
    const result = await DatabaseAgent.integrityCheck();
    if (!result.ok) {
      // Sentry に報告 + ユーザーへ「データの修復が必要」通知
      Sentry.captureException(new Error('DB integrity error'), { extra: result });
    }
    await AsyncStorage.setItem('lastIntegrityCheck', String(Date.now()));
  }

  // 月次: VACUUM（断片化解消）
  const lastVacuum = await AsyncStorage.getItem('lastVacuum');
  if (!lastVacuum || Date.now() - Number(lastVacuum) > 30 * 86400_000) {
    await DatabaseAgent.vacuum();
    await AsyncStorage.setItem('lastVacuum', String(Date.now()));
  }
}
```

### 11.8 マイグレーション戦略

| 環境                 | 実行タイミング                        | ツール                          |
| -------------------- | ------------------------------------- | ------------------------------- |
| Mobile（SQLite）     | アプリ起動時（自動）                  | expo-sqlite + カスタム migrator |
| Server（PostgreSQL） | デプロイ後（手動 or CD パイプライン） | `drizzle-kit migrate`           |

マイグレーションルール:

- **後方互換を保つ**: カラム追加は `ALTER TABLE ADD COLUMN ... DEFAULT` で実施。カラム削除は 2 フェーズ（アプリからの参照を先に除去 → 次のリリースで DROP）。
- **ゼロダウンタイム**: インデックス追加は `CREATE INDEX CONCURRENTLY`（PostgreSQL）。
- **ロールバック計画**: 各マイグレーションファイルに `down` 処理を記述。

### 11.9 クエリ安全性ルール

```typescript
// ✅ 正しい: Drizzle ORM の型安全クエリ
const recipes = await db.select().from(recipe).where(eq(recipe.familyId, familyId));

// ✅ 正しい: プリペアドステートメント
const stmt = db
  .select()
  .from(recipe)
  .where(eq(recipe.id, sql.placeholder('id')))
  .prepare('get_recipe');
const result = await stmt.execute({ id: recipeId });

// ❌ 禁止: 文字列結合による SQL 組み立て
const bad = `SELECT * FROM recipe WHERE familyId = '${familyId}'`; // SQL インジェクション
```

Drizzle ORM を経由することでパラメータは自動的にエスケープされる。生 SQL が必要な場合は `sql` タグテンプレートを使用し、コードレビューで必ずレビュアーが確認する。
