## 2. Google Play リリース（アプリ本体）

### 2-1. バージョンバンプ

`apps/mobile/app.json` の `version` と `android.versionCode` を上げる。
**app.json が唯一のソース**（`android/` は prebuild 生成物で gitignore。EAS は app.json から prebuild する）。

> **Android 自動バックアップの範囲**は `plugins/withDaidokoBackupRules.js` が制御している
> （クラウド自動バックアップ = SQLite＋設定のみで 25MB 枠に常時収まる / 端末間転送 = 写真込み全部。
> 写真のクラウド退避はアプリ内の SAF 書き出し・移行ファイル共有が担う — #79）。
> このプラグインや app.json を変えたら、ローカル検証ビルドは **`--prebuild` 必須**。

### 2-5. 本番構成の実機 E2E（マージ前検証の原則）

ローカル release ビルドは **必ず** `node scripts/agent/build-android.mjs` で行う
（`EXPO_NO_METRO_WORKSPACE_ROOT=1` を設定。生 gradlew は "Unable to resolve ./index.js" で失敗する）。

> **`EXPO_PUBLIC_*` を変えて連続ビルドするとき**（例: 広告オフ→オンの検証）は自動でキャッシュ破棄される。
> Gradle の bundleRelease タスクは JS ソースが不変だと前回のバンドルを再利用するため、フラグだけ変えても
> 反映されない実績があった（Pixel での広告テストで ADMOB*ENABLED=true が無視された）。
> build-android.mjs が `EXPO_PUBLIC*\*`の組をフィンガープリントとして記録し、変化していれば`createBundleReleaseJsAndAssets`の生成物を破棄してから叩く。手動で怪しいときは`apps/mobile/android/app/build/generated/{assets,res}/createBundleReleaseJsAndAssets` を消してもよい。

```powershell
adb reverse --remove-all     # localhost ブリッジを排除（本番 Railway に向ける）
adb install -r apps\mobile\android\app\build\outputs\apk\release\app-release.apk
# AI 機能（食べた/名寄せ）を操作 → railway logs で 200 を裏どり
```

## 5. トラブルシューティング（実際に踏んだもの）

| 症状                                                                                                 | 原因と対処                                                                                                                                                                                                                                                       |
| ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `eas submit` が "service account is missing the necessary permissions"                               | Play Console でサービスアカウントを招待していない → ユーザーと権限から招待（リリース権限）。反映に数分かかることあり                                                                                                                                             |
| "This release includes the com.google.android.gms.permission.AD_ID permission but your declaration…" | react-native-google-mobile-ads が AD_ID 権限を注入。広告未使用リリースでは `app.json` の `android.blockedPermissions` に `com.google.android.gms.permission.AD_ID` を入れて再ビルド。**広告を実際に出すリリースでは外し、Play の広告ID申告を「使用する」に変更** |
| 提出拒否後の versionCode                                                                             | バリデーション拒否では**未消費**。同じ versionCode で再提出できる                                                                                                                                                                                                |
| ローカル release ビルドが "Unable to resolve ./index.js"                                             | 生 gradlew 起因。`node scripts/agent/build-android.mjs` を使う                                                                                                                                                                                                   |
| pre-commit フックが Prettier で落ちる                                                                | 変更ファイルに CRLF/整形差分。`pnpm exec prettier --write <files>` してから commit                                                                                                                                                                               |
| Railway CLI が Unauthorized                                                                          | トークン失効。**ユーザー自身の対話ターミナル**で `railway login`（`!` 実行や browserless は非対話扱いで不可）                                                                                                                                                    |
| curl で日本語 POST が壊れる（Git Bash）                                                              | CP932 変換で入力が壊れ AI が空応答。`node -e "fetch(...)"` で送る                                                                                                                                                                                                |
| EAS ビルドがどのコードでビルドされるか                                                               | ローカル作業ディレクトリのアップロード（gitignore 準拠）。**main を checkout してから実行**                                                                                                                                                                      |

### 6-0-a. 収益化方針の決定（2026-07-06）— ★プライバシー制約

**当面「方針A = サブスク課金なし・広告(AdMob)＋BYOK のみ」で運用する。Play の販売アカウント（Google Payments merchant）は設定しない。**

理由 = **自宅住所の公開回避**。確定した制約:

- **個人アカウントで有料アプリ/アプリ内課金（サブスク含む）を有効化すると、自宅の完全な住所が Play 掲載の「デベロッパーについて」欄に自動公開される**（Google 公式ポリシー: 販売アカウントは完全住所の表示が必須）。**個人アカウントのままでは回避不可・有効化は不可逆**。
- 現状のストア公開情報（実地確認済み）= 氏名・メール・"Japan"（国のみ）。**番地は未公開**。
- サブスクを実現するには**組織アカウント化**が必要（開業届＋D-U-N-S取得＋バーチャルオフィス契約 約3週間。住所は非自宅にできる）。将来ユーザーが決断したら着手。
- **広告(AdMob)は Play 販売アカウント不要**（収益は AdMob 経由）。方針A なら住所公開は起きない。
- RevenueCat 設定（project `9b3a8171`・`premium`・Monthly Offering・公開キー `goog_diJqOsymwhIcMFTVKIYKjvhwPon`）は**温存**。将来 B を選べば eas.json にキー投入するだけで有効化できる。**当面 eas.json に RC キーは入れない**。

### 6-2. アプリ側の設定注入（リリース時に残っているもの）

AdMob の 2 ID は**配線済み**（上記）。広告・課金を有効にするリリースでは残り3点だけ:

1. `apps/mobile/eas.json` の `build.production.env` に追加（公開識別子でありコミット可）:
   ```json
   "EXPO_PUBLIC_ADMOB_ENABLED": "true",
   "EXPO_PUBLIC_REVENUECAT_API_KEY": "goog_…"
   ```
2. `apps/mobile/app.json` の `android.blockedPermissions` から `com.google.android.gms.permission.AD_ID` を**削除**
   （広告なしリリースのうちは残す — 外すと Play の広告ID申告「不使用」と衝突して提出拒否）
   - **注意（2026-07-10 実測）**: incremental prebuild は blockedPermissions への「追加」は反映するが
     「削除」は既存 AndroidManifest の remove 指定を消さない（サイレント no-op）。ローカル検証では
     生成済み `android/app/src/main/AndroidManifest.xml` から該当行を手で消すか `expo prebuild --clean`。
     EAS はゼロから生成するので影響なし
3. versionCode をバンプ → main リリース PR → EAS ビルド（§2 と同じ流れ）
   - ローカル検証ビルドは app.json のプラグイン変更を反映するため **`--prebuild` 必須**:
     `EXPO_PUBLIC_ADMOB_ENABLED=true node scripts/agent/build-android.mjs --prebuild`

### 6-3. Play Console 側の申告変更（AD_ID を含めるため必須・**3 点すべて独立項目**）

1. 「アプリのコンテンツ」→**広告**: 「アプリに広告を含める」= **はい**
2. 「アプリのコンテンツ」→**広告 ID**（`app-content/ad-id-declaration` — 広告・データセーフティとは**別の独立ページ**）:
   「広告 ID を使用する」= **はい**・用途=「広告、マーケティング」
   - **これを忘れると AAB アップロード自体が Google API に拒否される（2026-07-11 実測）**:
     eas submit が `This release includes the com.google.android.gms.permission.AD_ID permission but your declaration on Play Console says your app doesn't use advertising ID` で失敗。
     申告を保存した直後（審査送信前）に再実行したら通った — API 検証は保存済み申告を見る
3. **データセーフティ**を更新（§4 参照）:
   - 広告 ID: **収集する**・共有=広告パートナー（AdMob）・目的=広告またはマーケティング
   - 「デバイスまたはその他の ID」も同様に申告
4. 審査に送信（「公開の概要」の事前チェック完了後に自動送信される。申告と AAB の AD_ID 権限が一致しないと提出時に拒否される — §5 表参照）

### 6-4. 検証

- サンドボックス: Play の**ライセンステスター**に自分の Google アカウントを追加 → 内部テストトラックで実購入（テスト購入は課金されない）→ プレミアム反映・「購入を復元」を確認
- 広告: 本番ユニットでも**テストデバイス登録**すればテスト広告が出る。リワード視聴 → +1 回付与を確認
- UMP 同意フォームは GDPR 対象地域のみ表示（日本では出ない。エミュレータで確認する場合は AdsConsent のデバッグ設定で EEA を偽装）

## 7. iOS（App Store）リリース準備

Android 先行でリリース済み。iOS は Expo（RN）なので技術的には同一コードで出せるが、以下の準備が要る。
**方針決定（2026-07-06）: iOS 初回は「無料・広告なし・非取引者（non-trader）」で公開する。**

### 7-1. 機能パリティ（iOS での提供範囲）

| 機能                                        | 実装                      | iOS                                                                                                                  |
| ------------------------------------------- | ------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| 写真からレシピ（AI）                        | サーバー/BYOK 経由 Gemini | **提供**（`isNative` ゲートに変更済み。iOS も動く。端末内ラベリングは Android のみだがサーバー推論にフォールバック） |
| 食事写真→在庫消費（AI）                     | サーバー/BYOK 経由 Gemini | **提供**（画面ゲートなし）                                                                                           |
| 文字入り画像OCR / レシートOCR               | 端末内 ML Kit（Kotlin）   | **非提供**（iOS 版なし。add.tsx / pantry.tsx で入口を非表示）。将来 Apple Vision で iOS 実装可                       |
| ローカル機能・バーコード・URL/手動/テキスト | —                         | 提供                                                                                                                 |

- iOS の AI は「無料枠 1日1回 ＋ BYOK（自分の Gemini キーで無制限）」。**広告・プレミアム課金の導線は iOS では非表示**（非取引者方針。ペイウォールの「広告を見て」は広告 SDK 無効なら出ない）。
- サーバー AI 呼び出しコストはサーバー側 `INFER_GLOBAL_DAILY_LIMIT` で総額頭打ち。

### 7-2. プライバシー / DSA 取引者ステータス

- Apple も EU 向けに取引者/非取引者の申告が必須（未申告だと EU App Store から削除）。
- **無料・広告なし・趣味 → 非取引者 → 住所は非公開**（今回の方針）。
- 将来 iOS に広告を入れる場合は取引者になるが、**Apple は個人でも P.O. Box（私書箱）の使用を許可**（Play と違い自宅住所を出さずに済む）。

### 7-3. 前提（ユーザー作業）

- **Apple Developer Program 登録（年 $99）**。Apple ID＋支払い情報が必要（代行不可）。承認に1〜2日。
- App Store Connect でアプリ枠を作成（bundle id `com.daidoko.app` は app.json 設定済み）。

### 7-4. ビルド・提出（Apple 登録後・macOS で実行）

Mac 側の実作業手順は **`.claude/skills/ios-release`** に集約（シミュレータ確認 → スクショ → EAS ビルド → TestFlight → 提出）。要点:

- **iOS 固有作業は Mac で**（Windows では Xcode/シミュレータ不可）。シミュレータ動作確認 = `pnpm --filter mobile exec expo run:ios`。
- **App Store スクショは自動取得できる**（macOS）: `node scripts/release/capture-ios-screenshots.mjs`
  （`xcrun simctl` で 9:41・満充電固定 → ディープリンク → screenshot。出力 `docs/store/app-store/phone-screenshots/`）。
- EAS が iOS をクラウドビルド（**ローカル Mac ビルドは不要だが、シミュレータ確認・スクショに Mac が要る**）。署名は EAS が Apple ログインで自動管理。
- `eas.json` に iOS の production ビルド／submit プロファイルを追加（現状は android のみ）。`ITSAppUsesNonExemptEncryption: false` 設定済み。
- 広告 SDK: iOS は広告なしのため `react-native-google-mobile-ads` の iosAppId はテスト ID のまま同梱される。**クリーンにするなら iOS 向けに ads プラグインを app.config.js の条件分岐で除外する検討余地あり**（App Privacy を簡素化）。

### 7-5. 残るギャップ・注意

- **スクショのアップロード**は App Store Connect Web UI か fastlane deliver（Play のような API 一括反映スクリプトは未整備）。取得は §7-4 のスクリプトで自動化済み。
- **App Privacy（栄養ラベル）**: AI 機能利用時に写真・食材名をサーバー送信する旨を申告（Play のデータセーフティ相当）。端末内 OCR は iOS で無効化済みのため申告不要。
- **iOS の描画確認は Windows 環境では不可**。Mac のシミュレータ（§7-4）または EAS iOS ビルド後の TestFlight で検証する。
