## 1. サーバー（Railway）デプロイ

対象: `apps/server`（Hono）。本番 URL: `https://daidoko-production.up.railway.app`
サービス名: `daidoko` / プロジェクト: `daidoko`（production 環境）

```bash
# 1) 環境変数の確認（キーの値は表示しないこと）
railway variables --service daidoko --json > vars.json  # → キー名のみ確認して削除
# 必須: GEMINI_API_KEY（AI エンドポイント用）

# 2) デプロイ（リポジトリルートから。railway.json が apps/server/Dockerfile を指す）
railway up --service daidoko --detach

# 3) 完了確認（status が SUCCESS になるまで）
railway deployment list --service daidoko --json   # 先頭要素の .status

# 4) 疎通確認
curl https://daidoko-production.up.railway.app/health
# AI エンドポイント（日本語は Git Bash だと CP932 で壊れるので node fetch で確認）
node -e "fetch('https://daidoko-production.up.railway.app/api/v1/resolve/names',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({names:['たまご']})}).then(r=>r.json()).then(j=>console.log(JSON.stringify(j)))"

# 5) ログ確認
railway logs --service daidoko
```

**環境変数の追加・変更**（本番シークレット書き込みはユーザー承認必須）:


## 2. Google Play リリース（アプリ本体）

### 2-1. バージョンバンプ

`apps/mobile/app.json` の `version` と `android.versionCode` を上げる。
**app.json が唯一のソース**（`android/` は prebuild 生成物で gitignore。EAS は app.json から prebuild する）。

### 2-2. develop → main リリース PR

```bash
gh pr create --base main --head develop --title "release: x.y.z (versionCode NNNNN)"
gh pr checks <PR> --watch   # Quality Gates + Security Audit
gh pr merge <PR> --merge    # develop ブランチは削除しない
```

### 2-3. EAS production ビルド（main から）

```bash
git checkout main && git pull
cd apps/mobile
pnpm exec eas build -p android --profile production --non-interactive --no-wait
# ビルド ID をメモ → 状態ポーリング
pnpm exec eas build:view <BUILD_ID> --json   # .status が FINISHED / .artifacts.buildUrl
```

### 2-4. AAB 検証（提出前チェック）

```bash
curl -sL -o Temp/eas-aab-NNNNN/app-release.aab "<artifact URL>"
# 16KB ページアライメント（42 .so 全 PASS が期待値）
python C:/tmp/check_elf_align.py Temp/eas-aab-NNNNN/app-release.aab
```

### 2-5. 本番構成の実機 E2E（マージ前検証の原則）

ローカル release ビルドは **必ず** `node scripts/agent/build-android.mjs` で行う
（`EXPO_NO_METRO_WORKSPACE_ROOT=1` を設定。生 gradlew は "Unable to resolve ./index.js" で失敗する）。

```powershell
adb reverse --remove-all     # localhost ブリッジを排除（本番 Railway に向ける）
adb install -r apps\mobile\android\app\build\outputs\apk\release\app-release.apk
# AI 機能（食べた/名寄せ）を操作 → railway logs で 200 を裏どり
```

### 2-6. Play へ提出（CLI）

```bash
cd apps/mobile
pnpm exec eas submit -p android --profile production --path <AABパス> --non-interactive
# eas.json の submit.production に serviceAccountKeyPath / track: production 設定済み
```

`update-play-screenshots.mjs` の ORDER 配列。変えるときは両方更新する）。


## 5. トラブルシューティング（実際に踏んだもの）

| 症状                                                                                                 | 原因と対処                                                                                                                                                                                                                                                       |
| ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `eas submit` が "service account is missing the necessary permissions"                               | Play Console でサービスアカウントを招待していない → ユーザーと権限から招待（リリース権限）。反映に数分かかることあり                                                                                                                                             |
| "This release includes the com.google.android.gms.permission.AD_ID permission but your declaration…" | react-native-google-mobile-ads が AD_ID 権限を注入。広告未使用リリースでは `app.json` の `android.blockedPermissions` に `com.google.android.gms.permission.AD_ID` を入れて再ビルド。**広告を実際に出すリリースでは外し、Play の広告ID申告を「使用する」に変更** |
| 提出拒否後の versionCode                                                                             | バリデーション拒否では**未消費**。同じ versionCode で再提出できる                                                                                                                                                                                                |
| ローカル release ビルドが "Unable to resolve ./index.js"                                             | 生 gradlew 起因。`node scripts/agent/build-android.mjs` を使う                                                                                                                                                                                                   |
| pre-commit フックが Prettier で落ちる                                                                | 変更ファイルに CRLF/整形差分。`pnpm exec prettier --write <files>` してから commit                                                                                                                                                                               |
| Railway CLI が Unauthorized                                                                          | トークン失効。**ユーザー自身の対話ターミナル**で `railway login`（`!` 実行や browserless は非対話扱いで不可）                                                                                                                                                    |
| curl で日本語 POST が壊れる（Git Bash）                                                              | CP932 変換で入力が壊れ AI が空応答。`node -e "fetch(...)"` で送る                                                                                                                                                                                                |
| EAS ビルドがどのコードでビルドされるか                                                               | ローカル作業ディレクトリのアップロード（gitignore 準拠）。**main を checkout してから実行**                                                                                                                                                                      |

## 6. 広告・プレミアム課金の本番化リリース

設計は `docs/フリーミアム設計.md`。コード（無料枠・ペイウォール・RevenueCat 抽象・リワード広告・UMP 同意管理・BYOK）は実装済みで、
**既定ではすべて無効**（広告 UI 非表示・購入はスタブの「準備中」）。本番化はビルド時の設定注入だけで有効になる。

### 6-1. 前提（ストア側）

1. **AdMob — 完了（2026-07-05）**: だいどこ登録済み。
   - アプリ ID: `ca-app-pub-2633806931583277~2829239666`（app.json 配線済み）
   - リワードユニット ID: `ca-app-pub-2633806931583277/2276751496`（eas.json 配線済み）
   - 承認ステータス「要審査」→ 承認まで配信制限。**app-ads.txt** をデベロッパーサイトに要設置:
     `google.com, pub-2633806931583277, DIRECT, f08c47fec0942fa0`
     （ドメインは Play 掲載情報のウェブサイト欄と一致させる）
2. **RevenueCat（ユーザー作業→設定は自動化可）**: アカウント作成 → プロジェクト → Android の **public SDK key** 取得。
   エンタイトルメント `premium` と、月額サブスクを含む Offering（current / `monthly` パッケージ）を作成。
   Play 連携のサービス認証情報 JSON アップロードはユーザー作業（秘密鍵）
3. **Google Play（ユーザー作業）**: **Google Payments 販売アカウント**のセットアップ（収益化ページから。
   未設定だと定期購入ページ自体がロック）→「収益化」→ 定期購入商品を作成し、RevenueCat の Product に紐付け

### 6-2. アプリ側の設定注入（リリース時に残っているもの）

AdMob の 2 ID は**配線済み**（上記）。広告・課金を有効にするリリースでは残り3点だけ:

1. `apps/mobile/eas.json` の `build.production.env` に追加（公開識別子でありコミット可）:
   ```json
   "EXPO_PUBLIC_ADMOB_ENABLED": "true",
   "EXPO_PUBLIC_REVENUECAT_API_KEY": "goog_…"
   ```
2. `apps/mobile/app.json` の `android.blockedPermissions` から `com.google.android.gms.permission.AD_ID` を**削除**
   （広告なしリリースのうちは残す — 外すと Play の広告ID申告「不使用」と衝突して提出拒否）
3. versionCode をバンプ → main リリース PR → EAS ビルド（§2 と同じ流れ）
   - ローカル検証ビルドは app.json のプラグイン変更を反映するため **`--prebuild` 必須**:
     `EXPO_PUBLIC_ADMOB_ENABLED=true node scripts/agent/build-android.mjs --prebuild`

### 6-3. Play Console 側の申告変更（AD_ID を含めるため必須）

1. 「アプリのコンテンツ」→**広告**: 「アプリに広告を含める」= **はい**
2. **データセーフティ**を更新（§4 参照）:
   - 広告 ID: **収集する**・共有=広告パートナー（AdMob）・目的=広告またはマーケティング
   - 「デバイスまたはその他の ID」も同様に申告
3. 審査に送信（申告と AAB の AD_ID 権限が一致しないと提出時に拒否される — §5 表参照）

### 6-4. 検証

- サンドボックス: Play の**ライセンステスター**に自分の Google アカウントを追加 → 内部テストトラックで実購入（テスト購入は課金されない）→ プレミアム反映・「購入を復元」を確認
- 広告: 本番ユニットでも**テストデバイス登録**すればテスト広告が出る。リワード視聴 → +1 回付与を確認
- UMP 同意フォームは GDPR 対象地域のみ表示（日本では出ない。エミュレータで確認する場合は AdsConsent のデバッグ設定で EEA を偽装）
