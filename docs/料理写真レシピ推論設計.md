# 料理写真レシピ推論設計（Vision LLM）

> ステータス: ドラフト（実装前の設計合意用） / 起票 2026-06
>
> 対象機能: **料理の写真（＋任意の感想・文脈テキスト）から、編集前提のレシピ下書きを AI 推論で生成する**。
>
> 本書は `docs/レシピ作成フロー.md` の「料理写真推測」節を **heuristic 前提から Vision LLM 推論前提へ改訂する**ための設計提案である。

---

## 1. 背景と認識合わせ

### 1-1. 現状（実装済み）の「料理写真から推測」は heuristic

現行 `src/services/recipe-photo-inference.service.ts` の実体は、**ML Kit の画像ラベル → キーワードでプロファイルを当て → ほぼ空のテンプレを返す** ものである。

```
GENERIC_INGREDIENTS = ['主食材（写真を見て確認）', '野菜（写真を見て確認）', '調味料', '油またはバター']
GENERIC_STEPS       = ['写真から分かる食材を確認し…', '主食材に火が通るまで加熱し…', '味見をして…']
```

これは `docs/アーキテクチャ設計.md §7`（料理写真下書き推測＝クライアント/heuristic）および `docs/レシピ作成フロー.md`（「画像ラベルから低信頼の下書き」「100点の完成レシピを自動保存しない」）の**意図的な設計判断**であり、オフライン・端末内・低遅延を優先したもの。**料理そのものからレシピを推論してはいない。**

### 1-2. 本当に欲しい機能

> お店で食べた料理の写真 ＋（任意で）食べた感想・店名・気づき などをインプットに、**その料理を再現するためのレシピ（材料・分量・手順）を推論して下書き化する。**

これは画像ラベル（"food", "curry"）では原理的に不可能で、**マルチモーダル（Vision）LLM** のタスクである。よって本書は heuristic とは別系統の推論パスを新設する。

### 1-3. 2 つの「写真からレシピ」を明確に分離

| 機能                   | 入力                                         | 技術                | 実行場所             | 本書の対象               |
| ---------------------- | -------------------------------------------- | ------------------- | -------------------- | ------------------------ |
| **OCR（文字抽出）**    | レシピ本・手書きメモ・メニューの**文字**画像 | ML Kit テキスト認識 | 端末内               | 対象外（実装済み・維持） |
| **料理写真レシピ推論** | **料理そのもの**の写真＋任意テキスト         | **Vision LLM**      | サーバー経由クラウド | **本書の対象**           |

両者は入口（追加方法ピッカー）で別項目として残す。

## 2. スコープ

### 2-1. 機能要件

- 入力: 料理写真 1 枚（必須）＋ 自由テキスト（任意、例: 「○○屋の麻婆豆腐。痺れが強め、ひき肉多め」）。
- 出力: `RecipeFormData` 形の**下書き**（title / titleReading / description / servings / cookTimeMin / ingredients[] / steps[] / tags[]）＋ 推論信頼度＋根拠サマリ＋警告。
- 出力は**確定保存ではなく `RecipeForm` の初期値として遷移**する（既存方針を踏襲。ユーザーが必ず確認・編集してから保存）。
- 任意テキストは推論の文脈として優先的に反映する（味の方向性・地域・辛さ等）。

### 2-2. 非対象（本フェーズ）

- 動画・複数枚の同時推論（将来）。
- 栄養計算・原価計算。
- オフラインでの LLM 推論（端末内マルチモーダルは非現実的）。
- 推論結果の自動保存・自動共有。

---

## 3. 既存実装との関係（heuristic を fallback 化）

A2 エージェント `runRecipePhotoAgent(input, dependencies)` は依存注入で、`labelImage` / `recognizeText` / `inferRecipe` 等を差し替えられる。ここに **`inferRecipeFromVision`（サーバー推論 provider）を追加**し、優先順位を次のように定める。

```
1. Vision LLM 推論（オンライン＋同意済み＋画像に料理が写っている）  ← 新設・primary
2. 画像内に十分な文字 → OCR テキスト解析（既存）
3. heuristic ラベル下書き（オフライン/未同意/LLM失敗 時の fallback、既存）
```

`RecipePhotoAgentInput` を拡張:

```ts
export interface RecipePhotoAgentInput {
  imageUri: string;
  context?: string; // 任意の感想・店名・補足
  allowCloudInference?: boolean; // 同意状態（未同意なら Vision をスキップ）
}
```

`RecipePhotoAgentDependencies` に追加:

```ts
inferRecipeFromVision?: (args: {
  imageUri: string;
  context?: string;
}) => Promise<VisionRecipeInferenceResult>; // 失敗時 throw → 既存 fallback へ
```

> 既存テスト（`recipe-photo.agent.test.ts` 等）は依存注入なので、Vision provider をモックして追加ケースを足すだけで済む。heuristic 経路の回帰は起きない。

## 4. アーキテクチャ

端末上で Vision LLM は動かせないため、**Hono サーバー（Railway 稼働中）に推論エンドポイントを新設**し、サーバーがクラウド Vision LLM を呼ぶ。API キーは**サーバーのみ**が保持する（アプリに鍵を置かない）。

```
[Mobile]                         [Hono Server (Railway)]            [Vision LLM]
 import-photo.tsx
   └ A2 agent
       └ visionProvider ──HTTPS──> POST /api/v1/infer/photo ──API──> Claude / Gemini (vision)
            (resized JPEG          └ runPhotoInferAgent              └ structured JSON (tool use)
             base64 + context)        └ プロンプト整形・検証・整形
       <──── AgentResult<draft> ─────┘
   └ RecipeForm に prefill して遷移
```

データフロー要点:

- 画像は送信前に**端末で縮小**（長辺 ~1024px、JPEG）。`expo-image-manipulator` の既存 `ImagePreprocessAdapter` を再利用。送信量とコストを抑える。
- サーバーは LLM 応答を `recipeFormSchema` で検証してから返す（壊れた JSON はサーバーで弾く）。
- 既存の `AgentResult` パターン（常に 200、エラーは body）を踏襲。

## 5. API 設計

`apps/server/src/routes/infer.ts`（`routes/import.ts` を踏襲）。

### 5-1. リクエスト

```
POST /api/v1/infer/photo
Content-Type: application/json
```

```jsonc
{
  "imageBase64": "<縮小済みJPEGのbase64>", // 必須。最大 ~2MB を zValidator で制限
  "mimeType": "image/jpeg",
  "context": "○○屋の麻婆豆腐。痺れ強め", // 任意・最大 1000 文字
}
```

### 5-2. レスポンス（AgentResult<VisionRecipeInferenceResult>）

```jsonc
{
  "ok": true,
  "data": {
    "draft": {
      // recipeFormSchema 準拠
      "title": "四川風 麻婆豆腐",
      "titleReading": "しせんふう まーぼーどうふ",
      "description": "痺れと辛さを効かせた…",
      "servings": 2,
      "cookTimeMin": 20,
      "ingredients": [
        { "groupLabel": "", "name": "木綿豆腐", "amount": "1丁", "note": "" },
        { "groupLabel": "", "name": "豚ひき肉", "amount": "150g", "note": "" },
        // …
      ],
      "steps": [
        { "body": "豆腐を2cm角に切り、塩茹でして水切りする。" },
        // …
      ],
      "tags": ["中華", "辛い"],
    },
    "confidence": "medium", // high | medium | low
    "evidenceSummary": "写真から麻婆豆腐と判断。感想の『痺れ強め』を反映。",
    "warnings": ["分量・手順は写真からの推測です。必ず確認・調整してください。"],
  },
}
```

エラー時:

```jsonc
{ "ok": false, "error": { "code": "VISION_INFER_FAILED", "message": "…", "retryable": true } }
```

エラーコード（`@daidoko/shared` の `AgentErrorCode` に追加）: `VISION_INFER_FAILED` / `VISION_NOT_A_DISH`（料理が写っていない）/ `VISION_RATE_LIMITED` / `VISION_DISABLED`。

## 6. LLM 選定

| 観点                     | Claude（Anthropic, vision）           | Gemini（Google, vision）           |
| ------------------------ | ------------------------------------- | ---------------------------------- |
| 品質（料理の妥当な再現） | 高                                    | 高                                 |
| 日本語                   | 良                                    | 良                                 |
| 構造化出力               | tool use で堅牢                       | JSON mode                          |
| コスト                   | 中〜高                                | Flash 系は安価                     |
| 既存土台                 | `.env.example` に `ANTHROPIC_API_KEY` | `.env.example` に `GEMINI_API_KEY` |

**方針**: サーバー側を **provider 抽象**（`VisionRecipeProvider` interface）にして 1 実装から始め、env で切替可能にする。**MVP は Gemini Flash 系（低コスト・消費者アプリ向け）を既定**、品質要件が上がれば Claude（Sonnet 4.x 級）へ。いずれも**構造化出力（tool use / JSON schema）で `RecipeFormData` を強制**する。

> 最終的なモデル ID は実装時の最新を採用（本書では固定しない）。

## 7. プロンプト設計（方針）

- **システム**: 「あなたは料理写真からレシピを再現する日本語のシェフ。写真と任意の感想から、家庭で再現できる現実的な材料・分量・手順を推論し、指定 JSON で返す。不確実な点は無理に断定せず、一般的で安全な前提を置く。」
- **構造化出力**: `recipeFormSchema` 相当の JSON Schema を tool として渡し、それ以外を返させない。
- **任意テキストの扱い**: context があれば味の方向性・辛さ・地域・量を優先反映。矛盾時は写真を主、テキストを補正に。
- **不確実性**: 写真だけでは確定できない分量・加熱時間は「一般的な目安」を入れ、`warnings` に確認喚起を必ず付す。`confidence` を写真の鮮明さ・料理の特定しやすさで自己申告させる。
- **ガード**: 料理でない画像は `VISION_NOT_A_DISH` を返す（プロンプト＋サーバー判定）。
- **再現性/評価**: ゴールデン画像セット（10〜20 枚）で出力の妥当性を回帰評価（§12）。

## 8. モバイル統合

- `import-photo.tsx`（S10b）に**任意テキスト入力欄**（感想・店名）を追加。撮影/選択 → 縮小 → サーバー推論 → `RecipeForm` に prefill 遷移。
- 新 provider `src/services/vision-recipe.provider.ts`: 縮小画像＋context を `POST /api/v1/infer/photo` に送り `AgentResult` を返す。サーバー URL は既存の構成（URL import と同じ base）を再利用。
- A2 agent に `inferRecipeFromVision` を注入（§3）。`allowCloudInference` が false / オフライン / 失敗時は既存 herustic にフォールバックし、`warnings` に理由を出す。
- 既存の `PhotoCaptureService` / `ImagePreprocessAdapter` を再利用（重複実装しない）。

## 9. 同意・プライバシー（重要）

- **クラウド送信の明示同意が必須**。初回利用時に「この機能は写真を解析サーバー（および AI 提供元）に送信します」を明示し、オプトイン。設定にトグルを置く（既定 OFF か、初回ダイアログで ON）。
- **プライバシーポリシー更新**（現行 gist）: 画像の送信先・保存有無・目的を明記。サーバーは**画像を永続保存しない**（推論後破棄）方針を既定とし、ポリシーに記載。
- Google Play のデータ安全性フォーム更新（画像データの送信＝「写真」収集の申告が必要になる）。
- 未同意ユーザーには Vision を出さず、heuristic / OCR / 手動のみ提供。

## 10. セキュリティ・濫用対策

- **API キーはサーバーのみ**（`ANTHROPIC_API_KEY` / `GEMINI_API_KEY`）。アプリへ埋め込まない。
- 現状サーバーは認証/DB なし。LLM エンドポイントは課金を伴うため**最低限の濫用対策**を入れる:
  - 入力サイズ上限（base64 ~2MB、context 1000 文字）。
  - レート制限（IP / 端末トークン単位、メモリ or 軽量ストア）。
  - 将来的に端末トークン（匿名）で識別。
- CORS / オリジン制限、リクエストロギング（画像本体はログに残さない）。

## 11. コスト

- 1 推論 = Vision LLM 1 コール（画像 1 枚＋短文）。**画像縮小で入力トークンを抑制**。
- 既定 Gemini Flash 系なら 1 推論あたり概ね数円未満想定（モデル次第、実測で確定）。Claude 採用時は要試算。
- 上限/クォータ（1 端末あたり 1 日 N 回 等）を検討。無料枠 vs 将来課金は別途方針。

## 15. 決定事項

実装着手前の確定（2026-06 合意）。

1. **モデル**: ✅ **Gemini Flash 系**で MVP を構築（低コスト・消費者アプリ向け）。サーバー側 `VisionRecipeProvider` 抽象で将来 Claude へ切替可能にしておく。
2. **同意方式**: ✅ **初回オプトインダイアログ**。機能を初めて使うときにクラウド送信を説明して同意を取得。同意状態は永続化し、設定からも変更可能にする。
3. **画像保存**: ✅ **サーバーで推論後即破棄**（永続保存しない）。プライバシーポリシーに明記。
4. **コスト上限**: 〔仮確定〕**1 端末あたり 1 日 20 回**を既定上限（環境変数で調整可）＋ レート制限。運用しながら調整。
5. **リリース単位**: 〔仮確定〕**10008 とは別の次バージョン**（v1.2.0 / versionCode 10009 以降）。10008（16KB＋OCR）は先行リリース。
6. **heuristic の去就**: 〔仮確定〕**オフライン fallback として残す**（オフライン/未同意/LLM 失敗時に従来の下書きを返し、`warnings` に理由を表示）。

> 4〜6 は推奨に基づく仮確定。変更があれば本節を更新する。
