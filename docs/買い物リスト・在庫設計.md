# 買い物リスト・在庫（パントリー）設計

> 「家の在庫を核にした食材ライフサイクル」の設計。買い物リスト単体ではなく、
> **在庫 ⇄ 買い物リスト ⇄ レシピ** のループとして設計する。実装前に必ず参照。
> 関連: `docs/データ設計.md` / `docs/レシピ作成フロー.md` / `docs/フリーミアム設計.md`

---

## 1. 狙い

- 家にある食材（在庫）を把握し、**足りないものだけ買う／あるもので作る／なくなりそうを知る**を回す。
- 入力の手間を下げる仕掛け（レシピ材料の一括取り込み、レシート読み取り、将来は食事写真）。

## 2. 全体像（3つのループ）

```
        ┌──────────────┐   足りない材料を追加      ┌──────────────┐
        │   レシピ      │ ───────────────────────▶ │  買い物リスト  │
        │ (ingredients) │ ◀─ 在庫で作れる？ 判定 ── │  (shopping)   │
        └──────┬───────┘                           └──────┬───────┘
               │ 在庫で充足判定                            │ 「買った」→ 在庫へ
               ▼                                          ▼
        ┌─────────────────────────── 在庫 (pantry) ───────────────────────────┐
        │  家にある食材（数量・単位・残量しきい値）。残量低下で通知。           │
        │  入力: 手動 / 買い物リストから / レシート写真 / (将来)食事写真から推定 │
        └──────────────────────────────────────────────────────────────────────┘
```

## 3. 置き場所（UI / ナビゲーション）

- タブは現状の **4 つ（ホーム/レシピ/＋追加/設定）を維持**。
- **ホーム右上にカート🛒アイコン**（既存のカレンダー/ギャラリー アイコン横）→ 買い物リスト画面。
- 在庫画面へは、買い物リスト画面のヘッダ、または設定/ホームから。
- レシピ詳細に **「買い物リストに追加」**（足りない材料を一括追加）。

## 4. データモデル（端末ローカル SQLite / 家族グループ単位）

新規 2 テーブル。既存 `recipes/ingredients` は変更しない（突合は名前正規化で行う）。

### pantry_items（在庫）

| 列                     | 型      | 意味                               |
| ---------------------- | ------- | ---------------------------------- |
| id                     | TEXT PK |                                    |
| family_id              | TEXT    | 家族グループ                       |
| name                   | TEXT    | 表示名（例: 玉ねぎ）               |
| name_normalized        | TEXT    | 突合キー（正規化済み, §6）         |
| quantity               | REAL?   | 数量（任意。未管理なら null）      |
| unit                   | TEXT?   | 単位（個・g・ml など。任意）       |
| low_stock_threshold    | REAL?   | この数量以下で「なくなりそう」通知 |
| updated_at, created_at | TEXT    |                                    |

### shopping_items（買い物リスト）

| 列              | 型      | 意味                                          |
| --------------- | ------- | --------------------------------------------- |
| id              | TEXT PK |                                               |
| family_id       | TEXT    | 家族グループ                                  |
| name            | TEXT    | 表示名                                        |
| name_normalized | TEXT    | 突合キー                                      |
| amount          | TEXT?   | 分量メモ（例: 「2個」「大さじ1」自由記述）    |
| checked         | INTEGER | 0/1                                           |
| source          | TEXT    | 'manual' / 'recipe' / 'low_stock' / 'receipt' |
| recipe_id       | TEXT?   | 由来レシピ（source='recipe' のとき）          |
| sort_order      | INTEGER |                                               |
| created_at      | TEXT    |                                               |
| checked_at      | TEXT?   | チェック時刻（在庫反映の判断に使用）          |

> マイグレーション: `migrate.ts` の `CREATE_TABLES_SQL` に 2 テーブルを追記
> （`CREATE TABLE IF NOT EXISTS` なので既存 DB にも安全に追加される）。

## 5. 機能別設計

### 5.1 買い物リスト（P1）

- 手動で品目追加（name ＋ 任意 amount）。チェック（永続）。並べ替え・削除・**チェック済み一括削除**。
- レシピ詳細「買い物リストに追加」: そのレシピの材料を `source='recipe'`, `recipe_id` 付きで追加。
- 重複: 同名（正規化）が未チェックで既にあれば**重複追加しない**（amount は結合しない＝単位が多様なため）。

### 5.2 在庫（pantry, P2）

- 一覧・追加・編集（数量・単位・しきい値）・削除。
- 買い物リストで**チェック→「在庫に入れる」**動線（その品目を pantry へ upsert）。

### 5.3 レシピの「足りない材料」→ 買い物リスト（P2）

- レシピ材料（正規化）のうち **pantry に無い／数量がしきい値以下**のものを抽出し、買い物リストへ一括追加。
- 数量の厳密比較はしない（単位が揃わないため、MVP は「在庫に存在するか」で判定）。

### 5.4 在庫で作れるレシピ（P4）

- 各レシピの材料に対する **pantry 充足率**（= 在庫にある材料数 / 全材料数）でランキング。
- 「ほぼ作れる（充足率高 ＋ 不足は調味料程度）」を上位表示。

### 5.5 残量通知（P3）

- pantry の `quantity <= low_stock_threshold` の品目を検出 → **ローカル通知**（先日実装の `notification.service` を流用）。
- トリガ: アプリ起動時／在庫更新時にチェックし、必要なら通知をスケジュール（1日1回まとめる等、頻度は要調整）。

### 5.6 レシート登録（P5）

- レシート写真 → **Gemini Vision の構造化出力**で `[{name, quantity?, unit?}]` を抽出 → pantry へ upsert。
- 代替: 端末内 OCR（ML Kit）＋パース（安価・精度低）。まず Vision で実装し、精度を見て調整。
- コスト: AI 呼び出し＝フリーミアム枠の扱いを決める（§7 未決）。

### 5.7 毎食写真 → 消費推定（P6・実験的）

- 食事写真 → 料理推定 → 典型材料を推定 → pantry を概算で減らす。
- **精度は未知**。あくまで補助（自動で減らしすぎない・確認を挟む）。opt-in。研究フェーズとして最後。

## 6. 食材名の正規化・突合（肝）

レシピ材料・在庫・レシート・手動入力を**名前で突合**するため、正規化が要。

- 既存の**かなユーティリティ**（`utils/kana`）を流用：全角/半角・カナ/かな統一、トリム、空白除去。
- 分量や修飾を除去（例: 「玉ねぎ 1個」→「玉ねぎ」、「みじん切り玉ねぎ」→ 要検討）。
- MVP は**完全一致（正規化後）**。表記ゆれ辞書（玉ねぎ＝玉葱＝たまねぎ）は段階的に。
- 数量・単位の自動合算は**しない**（MVP）。在庫は手動数量管理、買い物リストは品目ベース。

## 7. 未決事項（要相談）

1. **数量・単位**: MVP は「在庫に在る/無い」中心で、数量は任意管理。自動合算なしで良いか。
2. **AI コスト**: レシート/食事写真の AI 呼び出しを、写真レシピと同じ無料枠に含めるか、別枠/プレミアム特典にするか。
3. **共有範囲**: 在庫・買い物リストは**家族グループ共有**（family_id 単位）で良いか（同期は未実装なので当面は端末ローカル）。
4. **表記ゆれ辞書**: 食材マスタ（正規化辞書）を持つか、当面は単純正規化のみか。

## 8. フェーズ計画（1つずつ実装→テスト→エミュレーター確認→PR→マージ）

| Phase  | 範囲                                                         | 主な追加                                                                                          |
| ------ | ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| **P1** | 買い物リスト（手動＋レシピ材料＋チェック永続＋ホームカート） | `shopping_items`, `shopping-list.service`, 買い物リスト画面, ホームのカート, レシピ詳細の追加動線 |
| **P2** | 在庫 ＋ 「買った→在庫」＋「足りない材料→買い物リスト」       | `pantry_items`, `pantry.service`, 在庫画面, 突合ロジック                                          |
| **P3** | 残量通知                                                     | しきい値検出＋ローカル通知（流用）                                                                |
| **P4** | 在庫で作れるレシピ                                           | 充足率ランキング                                                                                  |
| **P5** | レシート登録 → 在庫                                          | Vision 構造化抽出                                                                                 |
| **P6** | 毎食写真 → 消費推定（実験）                                  | AI 推定＋確認フロー                                                                               |
