# 買い物リスト・在庫（パントリー）設計

> 「家の在庫を核にした食材ライフサイクル」の設計。買い物リスト単体ではなく、
> **在庫 ⇄ 買い物リスト ⇄ レシピ** のループとして設計する。実装前に必ず参照。
> 関連: `docs/データ設計.md` / `docs/レシピ作成フロー.md` / `docs/フリーミアム設計.md`

---

## 1. 狙い

- 家にある食材（在庫）を把握し、**足りないものだけ買う／あるもので作る／なくなりそうを知る**を回す。
- 入力の手間を下げる仕掛け（レシピ材料の一括取り込み、レシート読み取り、将来は食事写真）。

## 2. 全体像（3つのループ）

```
        ┌──────────────┐   足りない材料を追加      ┌──────────────┐
        │   レシピ      │ ───────────────────────▶ │  買い物リスト  │
        │ (ingredients) │ ◀─ 在庫で作れる？ 判定 ── │  (shopping)   │
        └──────┬───────┘                           └──────┬───────┘
               │ 在庫で充足判定                            │ 「買った」→ 在庫へ
               ▼                                          ▼
        ┌─────────────────────────── 在庫 (pantry) ───────────────────────────┐
        │  家にある食材（数量・単位・残量しきい値）。残量低下で通知。           │
        │  入力: 手動 / 買い物リストから / レシート写真 / (将来)食事写真から推定 │
        └──────────────────────────────────────────────────────────────────────┘
```

## 3. 置き場所（UI / ナビゲーション）

- タブは現状の **4 つ（ホーム/レシピ/＋追加/設定）を維持**。
- **ホーム右上にカート🛒アイコン**（既存のカレンダー/ギャラリー アイコン横）→ 買い物リスト画面。
- 在庫画面へは、買い物リスト画面のヘッダ、または設定/ホームから。
- レシピ詳細に **「買い物リストに追加」**（足りない材料を一括追加）。

## 4. データモデル（端末ローカル SQLite / 家族グループ単位）

新規 2 テーブル。既存 `recipes/ingredients` は変更しない（突合は名前正規化で行う）。

### pantry_items（在庫）

| 列                     | 型      | 意味                                |
| ---------------------- | ------- | ----------------------------------- |
| id                     | TEXT PK |                                     |
| family_id              | TEXT    | 家族グループ                        |
| name                   | TEXT    | 表示名（例: 玉ねぎ）                |
| name_normalized        | TEXT    | 突合キー（正規化済み, §6）          |
| quantity               | REAL?   | 数量（任意。未管理なら null）       |
| unit                   | TEXT?   | 単位（個・g・ml など。任意）        |
| low_stock_threshold    | REAL?   | この数量以下で「なくなりそう」通知  |
| jan_code               | TEXT?   | バーコード(JAN)。包装商品の厳密識別 |
| updated_at, created_at | TEXT    |                                     |

### shopping_items（買い物リスト）

| 列              | 型      | 意味                                          |
| --------------- | ------- | --------------------------------------------- |
| id              | TEXT PK |                                               |
| family_id       | TEXT    | 家族グループ                                  |
| name            | TEXT    | 表示名                                        |
| name_normalized | TEXT    | 突合キー                                      |
| amount          | TEXT?   | 分量メモ（例: 「2個」「大さじ1」自由記述）    |
| checked         | INTEGER | 0/1                                           |
| source          | TEXT    | 'manual' / 'recipe' / 'low_stock' / 'receipt' |
| recipe_id       | TEXT?   | 由来レシピ（source='recipe' のとき）          |
| sort_order      | INTEGER |                                               |
| created_at      | TEXT    |                                               |
| checked_at      | TEXT?   | チェック時刻（在庫反映の判断に使用）          |

> マイグレーション: `migrate.ts` の `CREATE_TABLES_SQL` に 2 テーブルを追記し、
> `CURRENT_SCHEMA_VERSION` を 2→3 に上げる（`CREATE TABLE IF NOT EXISTS` なので既存 DB にも安全に追加される）。
> Drizzle 定義（`schema.ts`）は既存どおり object-style の index 設定で追加する。

## 5. 機能別設計

### 5.1 買い物リスト（P1）

- 手動で品目追加（name ＋ 任意 amount）。チェック（永続）。並べ替え・削除・**チェック済み一括削除**。
- レシピ詳細「**材料を買い物リストに追加**」: そのレシピの材料を `source='recipe'`, `recipe_id` 付きで追加。
  **P1 は全材料を追加**（「足りない材料だけ」は pantry が要る **P2**）。
- 重複: 同名（正規化）が未チェックで既にあれば**重複追加しない**（amount は結合しない＝単位が多様なため）。
- 既存 per-recipe `shop.tsx`（このレシピ用・セッション限りのチェック、永続化なし）とは**別物**として残し、ラベルで区別する（統合は後フェーズ §8）。

### 5.2 在庫（pantry, P2）

- 一覧・追加・編集（数量・単位・しきい値）・削除。**数量は厳密管理（同一商品は合算）**。
- 買い物リストで**チェック→「在庫に入れる」**動線（その品目を pantry へ upsert／数量加算）。
- **バーコード(JAN)入力**: expo-camera で JAN をスキャン → 既知 JAN なら名前/単位を自動補完、未知なら**初回だけ名前入力 → `jan_code→名前/単位` をローカル記憶**（外部DB不要、使うほど賢くなる）。包装商品の厳密識別＋安全な合算に使う。

### 5.3 レシピの「足りない材料」→ 買い物リスト（P2）

- レシピ材料（正規化）のうち **pantry に無い／数量がしきい値以下**のものを抽出し、買い物リストへ一括追加。
- 数量の厳密比較はしない（単位が揃わないため、MVP は「在庫に存在するか」で判定）。

### 5.4 在庫で作れるレシピ（P4）

- 各レシピの材料に対する **pantry 充足率**（= 在庫にある材料数 / 全材料数）でランキング。
- 「ほぼ作れる（充足率高 ＋ 不足は調味料程度）」を上位表示。

### 5.5 残量通知（P3）

- pantry の `quantity <= low_stock_threshold` の品目を検出 → **ローカル通知**（先日実装の `notification.service` を流用）。
- トリガ: アプリ起動時／在庫更新時にチェックし、必要なら通知をスケジュール（1日1回まとめる等、頻度は要調整）。

### 5.6 レシート登録（P5）

- **既定＝端末内 OCR（無料）**：レシート写真 → `withDaidokoOcr`/ML Kit で文字列化 → **ヒューリスティック解析**で `[{name, quantity?, unit?}]` → pantry へ upsert。難所は OCR ではなく**レシート文字列→品目への解析**（略称・価格行・店コード）。
- **任意＝Vision 高精度モード（プレミアム特典）**：messy なレシートを Gemini の構造化出力で精度よく抽出。

### 5.7 毎食写真 → 消費推定（P6・実験的・プレミアム特典）

- 食事写真 → 料理推定 → 典型材料を推定 → pantry を概算で減らす。**Vision＝プレミアム特典**。
- **精度は未知**。あくまで補助（自動で減らしすぎない・確認を挟む）。opt-in。研究フェーズとして最後。

## 6. 食材名の正規化・突合（肝）

レシピ材料・在庫・レシート・手動入力を**名前で突合**するため、正規化が要。

- 正規化ヘルパ `normalizeItemName` を新設：`String.normalize('NFKC')`（全/半角統一）＋ 既存 `normalizeKana`（カナ→かな・小文字・trim）＋ 連続空白除去。
  ※ `normalizeKana` 単体は全/半角を統一しないため NFKC を併用する。
- 分量や修飾を除去（例: 「玉ねぎ 1個」→「玉ねぎ」、「みじん切り玉ねぎ」→ 要検討）。
- MVP は**完全一致（正規化後）**。表記ゆれ辞書（玉ねぎ＝玉葱＝たまねぎ）は段階的に。
- **包装商品は JAN コードで厳密識別**（名前突合より優先）。レシピ材料・バラ売りは JAN を持たないため名前正規化で突合（**JAN と名前正規化を併用**）。
- **数量は厳密管理＝同一商品は合算**（JAN 一致 or 同名(正規化)×同単位）。異なる単位は合算しない。

## 7. 決定事項（確定）

1. **数量・単位＝厳密管理**: 数量×単位を保持し、**同一商品は数量を合算**（**JAN 一致**、または 同名(正規化)×同単位）。異なる単位は合算しない。
2. **AI コスト**:
   - **食事写真→消費推定（Vision）＝プレミアム特典**。
   - **レシート＝端末内 OCR で無料**（既存 `withDaidokoOcr`/ML Kit 日本語）。任意で **Vision 高精度モード＝プレミアム特典**。
3. **共有範囲＝家族グループ単位**（`family_id`）。同期は未実装のため当面は端末ローカル。
4. **表記ゆれ＝単純正規化のみ**（かな/全半角統一＋トリム＋完全一致）。同義語辞書は後追い。**包装商品は JAN コードで厳密識別**。

## 8. フェーズ計画（1つずつ実装→テスト→エミュレーター確認→PR→マージ）

## 9. 既存システムとの整合（俯瞰チェック結果）

- **規約準拠**: `family_id`＝`getCurrentFamily().id`、id＝`generateId()`、時刻＝ISO 文字列、`isNativePlatform` ガード（web は空返し・書込 no-op）。FTS は不要で `recipe_fts` と干渉しない。
- **レシピ材料の取得**: `getRecipeDetail(id).ingredients`（`{ name, amount?, groupLabel?, note? }[]`）を流用。
- **既存 `shop.tsx` との関係**: レシピごと・セッション限りのチェック（永続化なし）。P1 の集約リストとは別物として残し、レシピ詳細では「材料を買い物リストに追加」（集約・永続）と区別。
- **将来の同期**: family 単位だが cloud 同期は未実装。同期実装時に `shopping_items`/`pantry_items` を `sync_meta`（vector clock）連携対象に含める。
- **既知の小修正**: `ai-key` 画面の `_layout` `href:null` 漏れ（PR #12）を P1 で同時修正。
