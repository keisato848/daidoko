## 2. Claude フック（`.claude/settings.json` で配線）

### PreToolUse — `scripts/agent/hook-pretool-guard.mjs`（Bash / PowerShell 対象）

実際に事故った・時間を失った操作だけをルール化する方針。

| 判定 | 対象                                                               | 理由                                                                         |
| ---- | ------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| deny | `git reset --hard` / `git checkout -- `                            | 作業消失                                                                     |
| deny | `adb uninstall` / `pm clear`                                       | 端末のローカルデータ（SQLite・写真）消去                                     |
| deny | 生 `gradlew assembleRelease/bundleRelease`                         | `EXPO_NO_METRO_WORKSPACE_ROOT` 未設定で必ず失敗 → `build-android.mjs` へ誘導 |
| ask  | `railway variables`（`--json` なし）                               | テーブル表示がシークレット値を会話ログに晒す                                 |
| ask  | `railway up` / `eas submit`                                        | 本番デプロイ・ストア提出（ユーザー明示承認の確認）                           |
| ask  | Bash ツールでの `adb` + `/sdcard`                                  | Git Bash のパス変換で壊れる → PowerShell へ誘導                              |
| ask  | `adb install`（`-r` なし） / `git push --force`（with-lease 除く） | データ保持・履歴保護                                                         |
| deny | `git push` で main を対象                                          | main 直 push 禁止（PR 経由のみ — CLAUDE.md）                                 |
| ask  | `cd apps/mobile && pnpm install/add`                               | hoisted ワークスペースのハイジャック防止（install はルートから）             |
| ask  | `eas build --profile production`                                   | EAS はローカルディレクトリをアップロード — main checkout の規約確認          |
| ask  | `gh pr merge` / `git merge`                                        | マージ前実機確認の規約（CLAUDE.md §5）                                       |

**注意**: ペイロードは snake_case（`tool_name` / `tool_input`）。settings.json の変更はセッション中でも即反映される。
ルール単体テストはトリガー文字列をコマンドラインに含めない（自分自身が deny される）— JSON ファイル経由で stdin に流す。

### PreToolUse — `scripts/agent/hook-pretool-edit-guard.mjs`（Edit / Write / MultiEdit 対象）

コマンドではなく**ファイルへの書き込み内容**を検査する（コミット前の scan-staged-secrets より手前の一次防衛）。

| 判定 | 対象                                                             | 理由                                                                                    |
| ---- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------- |
| deny | シークレットらしきリテラル（PEM 秘密鍵 / Google API キー）の書込 | 値は .env / C:\secure 等へ。許可先（.env\* / settings.local.json / リポジトリ外）は除外 |
| ask  | eas.json への `EXPO_PUBLIC_REVENUECAT_API_KEY` 追加              | 収益化方針A（課金保留）の間は投入禁止 — リリース手順 §6-0-a                             |
| ask  | package.json への `overrides` / `patchedDependencies` 追加       | SDK51 回避策の再導入防止（16KB 対応を壊すリスク）                                       |

### PostToolUse — `scripts/agent/hook-posttool-docs-reminder.mjs`（Edit / Write 対象）

**ドキュメント自律更新の1段目**。ドキュメント連動ファイル（対応表: `scripts/agent/lib/docs-map.mjs`）を編集した直後に「対応する手順書 / Skill / 設計書を同じブランチで更新せよ」というコンテキストを注入する。

| 変更ファイル                                                     | 更新を促す先                                        |
| ---------------------------------------------------------------- | --------------------------------------------------- |
| `scripts/agent/hook-*` / `.githooks/` / `.claude/settings.json`  | docs/開発ハーネス.md                                |
| `scripts/release/**`                                             | docs/リリース手順.md §3・update-store-listing Skill |
| `railway.json` / `apps/server/Dockerfile`                        | docs/リリース手順.md §1・deploy-server Skill        |
| `apps/mobile/eas.json`                                           | docs/リリース手順.md §2・release-play Skill         |
| `.claude/agents\|skills/**`                                      | docs/開発ハーネス.md §1 構成マップ                  |
| `apps/server/src/routes/**`                                      | docs/アーキテクチャ設計.md（API 一覧）              |
| `apps/mobile/src/db/schema\|migrate.ts`                          | docs/データ設計.md                                  |
| `apps/mobile/app.json`                                           | docs/リリース手順.md §2-1 / §6（AdMob・AD_ID）      |
| `apps/mobile/plugins/**`                                         | docs/リリース手順.md（--prebuild 必須）             |
| `docs/privacy-policy.md`                                         | gist 同期（リリース手順 §4）＋データセーフティ整合  |
| `.github/(skills\|agents\|hooks\|prompts)/**`                    | docs/開発ハーネス.md §1 構成マップ                  |
| `apps/mobile/src/services/(usage\|entitlement\|ad-reward\|byok)` | docs/フリーミアム設計.md                            |

### Stop — `scripts/agent/hook-stop-docs-guard.mjs`

**ドキュメント自律更新の2段目（強制力）**。ターン終了時に、連動ファイルが変更されているのに対応ドキュメント / Skill が一切未更新なら**停止を一度だけブロック**し、更新（または「更新不要」の理由明示）を実行させる。

- 検査範囲: 作業ツリー＋（feature ブランチなら）`develop...HEAD` の差分
- docs / `.claude/skills|agents` / CLAUDE.md のいずれかが同時に変更済みなら督促しない
- `stop_hook_active` フラグで再ブロックしない（無限ループ防止）

### SessionStart — `scripts/agent/hook-session-start.mjs`

`preflight.mjs --json` の結果（Node / pnpm / ADB 等の環境状態）をセッション冒頭に注入。

### 未配線（opt-in） — `scripts/agent/hook-posttool-validate.mjs`

編集ごとにスライス検証（lint/typecheck）を回すためセッションが重くなりすぎる。品質は pre-commit と CI で担保する。必要なら settings.json に追加。

## 3. Git フック（`.githooks/`）

**pre-commit**（順に実行・いずれか失敗でコミット中止）:

1. `scan-staged-secrets.mjs` — 追加行の PEM 秘密鍵 / Google API キー / サービスアカウント JSON を遮断
2. `format-staged.mjs` — ステージ済みファイルを Prettier で**自動整形して再ステージ**（整形差分での失敗ループを根絶）
3. `validate-changed-slice.mjs --staged` — 変更パッケージの prettier check / lint / typecheck
4. `test-customizations.mjs`

初回セットアップ: `node scripts/agent/install-git-hooks.mjs`（`core.hooksPath` を設定）。

## 4. 実機検証の規約（要点）

詳細は `.claude/agents/android-verifier.md` の「実機操作の実務規約」。

- adb は **PowerShell ツール**で（Git Bash は `/sdcard` を壊す）
- スクショは `pwsh scripts/agent/device-shot.ps1`（縮小込み・ロック検知つき）
- `svc power stayon true` は作業後に必ず `false`
- セキュアロックは adb で解除不可 → ユーザーに依頼
- 座標タップは直前スクショで確認（UI 変更でずれる）
- かな入力はローマ字合成で自動化可・漢字はユーザー依頼

## 5. ガードレールの多層構造

1. **Claude フック（PreToolUse）** — リポジトリ固有の既知事故の遮断・確認
2. **Claude Code 権限モード / auto-mode classifier** — 本番シークレット書き込み・本番デプロイ等の横断ガード（フックと重複してよい。フックは「理由の言語化」、classifier は「最後の砦」）
3. **git pre-commit** — シークレット・品質のコミット前ゲート
4. **CI（GitHub Actions）** — Quality Gates / Security Audit（LF 環境での最終判定。ローカル CRLF ノイズはここで白黒つく）
5. **リポジトリ規約（CLAUDE.md）** — マージ前実機確認・main 直 push 禁止

## 6. 秘密情報の取り扱い

- `GEMINI_API_KEY`: `apps/server/.env`（gitignore）と Railway 環境変数のみ。**会話・コミット・コマンドラインに値を出さない**（シェル変数経由で渡す）
- Play サービスアカウント鍵: `C:\secure\play-service-account.json`（リポジトリ外）
- 確認時は `--json` をファイルへ → キー名のみ列挙 → 即削除

## 7. エージェント編成とタスク分担

複数のワークストリーム（Android/Play・iOS/App Store・サーバー・ストア運用・調査）を、役割の異なる
サブエージェント（`.claude/agents/`）に分担する。メインループが**司令塔**として計画・承認ゲート・
外向きアクション（`eas submit` / `railway up` / 公開アップロード / データセーフティ）を担い、
各エージェントは**非破壊の検証・準備・調査**に徹して結果をメインループへ返す。

### 7-1. 役割分担（誰が何を持つか）

| エージェント           | 環境         | 担当                                                                | 外向きアクション                                       |
| ---------------------- | ------------ | ------------------------------------------------------------------- | ------------------------------------------------------ |
| `repo-research`        | 任意         | コード探索・実装面のマッピング（読み取り専用）                      | なし                                                   |
| `android-verifier`     | Windows/実機 | Android の preflight・端末健全性・署名・E2E トリアージ              | なし                                                   |
| `release-orchestrator` | Windows/実機 | Android のローカル build-install-verify ループ（開発検証）          | なし（`eas submit` はメインループ）                    |
| `ios-release-mac`      | **macOS**    | iOS シミュレータ確認・iOS スクショ・EAS iOS ビルド・TestFlight 準備 | なし（提出はメインループ＋承認）                       |
| `store-ops`            | 任意         | 掲載文・スクショの整備/検証/`--dry-run`                             | なし（公開反映・データセーフティはメインループ＋承認） |
| （司令塔）メインループ | —            | 計画・タスク割当・承認ゲート・**外向きアクション**                  | `eas submit` / `railway up` / 掲載公開 / DSA 申告      |

### 7-2. タスク管理: GitHub Issues が正、セッション Task ツールが作業中スプリント

**永続タスクボード = GitHub Issues（keisato848/daidoko）**。無料・認証済み（gh CLI）・Windows/Mac の
どのセッションからも同じボードが見え、公式 GitHub MCP サーバーでも操作できる。

- **担当** = `agent:*` ラベル（上表のエージェント名 / `agent:user` = ユーザー本人の作業 / `agent:main-loop` = 司令塔）
- **待ち要因** = `blocked:external`（Google/Apple 審査）・`blocked:decision`（ユーザーの方針判断）
- **トラック** = `track:monetization` / `track:ios` / `track:feature` / `track:harness` / `track:maintenance`
- **マイルストーン** = M1: 広告有効化リリース / M2: iOS 初回リリース / M3: バックログ
- 依存関係は Issue 本文の「前提: #NN」で表現。着手時に `gh issue edit` でセルフアサインし、完了時にクローズ

セッション内の **Task ツール**（TaskCreate/Update/List）は「いま進行中のスプリント」の追跡に使い、
着手時に該当 Issue 番号をタスク説明へ記す（セッション終了で消えるため、恒久情報は Issue へ）。

- **委譲の原則**: 外部待ちで動けないタスクにエージェントを起動しない（トークンの無駄）。着手可能になってから委譲する。
- **承認ゲート**: エージェントは外向きアクションの**直前で必ず止め**、メインループがユーザー承認を取ってから実行する。

### 7-3. パイプライン（Skill の連結）と保存ワークフロー

代表的な一連のワークフロー:

- **Android リリース**: `release-readiness`（保存ワークフローで事前監査）→ `release-verify`（成果物検証）→ `release-play`（提出）
- **広告有効化**: `monetize-golive` = console-browser-ops（申告）＋ emulator-verify（E2E）＋ release-play（提出）の連結
- **iOS リリース**: `ios-release`（macOS。ios-release-mac エージェントが実務、提出は司令塔）
- **ストア掲載更新**: store-ops（整備・dry-run）→ 司令塔（承認・公開反映 = update-store-listing）
- **障害時**: `incident-rollback`

`release-readiness` は `.claude/workflows/release-readiness.js`（Workflow ツールの名前指定で実行可能）。
